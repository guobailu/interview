# HRBP(不带案例) Q&A

## 1. 你工作履历中不停换业务域，每个业务域都得不到深耕，你怎么看
说实话我也反思过这个问题。但换个角度看，每次换业务域其实都是在拓宽我的能力边界。从海康的IoT停车场到嘀嘀的企业信息化再到天猫好房的电商商品，看起来跨度大，但底层的技术能力是通用的——高并发、分布式、系统设计这些能力是跨业务域积累的。而且正因为经历过不同业务，我对新业务的理解速度很快，到天猫好房两个月就能主导商品中台重构。我现在也想找一个能长期深耕的业务方向，电商到家这个赛道我很看好，希望能扎下来做深。

## 2. 如果在工作过程中，遇到了悬而不决的问题，怎么处理?
我的做法是先明确卡在哪，是信息不够、资源不够还是决策权不够。如果是信息不够，我会主动去找相关方对齐，拉个会把问题摆到台面上讨论，不让它一直悬着。如果是需要上级拍板的，我会把问题整理清楚，列出几个方案和各自的利弊，带着方案去找领导决策，不是空手去问怎么办。如果是跨团队协作卡住了，我会主动推动，必要时升级到双方leader层面。核心原则就是不等不靠，主动推进，设个deadline，到时间必须有结论。

## 3. 如果在工作过程中，和其他人有分歧或者意见不合怎么去解决？
先听对方的理由，搞清楚分歧的根源是什么。很多时候不是方案对错的问题，是大家站的角度不同，信息不对称。我会先把双方的出发点和顾虑摆出来，看能不能找到兼顾的方案。如果确实是技术方案的分歧，就拿数据和事实说话，做个对比分析或者小范围验证，让结果来决定。如果是优先级或资源的分歧，就拉上leader一起对齐目标，从业务价值角度排优先级。实在达不成一致的，我会尊重最终决策者的判断，先执行再看效果。不会因为意见不合就消极对待，毕竟目标是一致的。

## 4. 如何在压力大的情况，保证自己的工作效率、质量？
压力大的时候最怕的是乱，所以我会先理清优先级。把手上的事按紧急重要程度排序，P0的先搞定，P2的该砍就砍，不要什么都想做。然后拆解任务，大任务拆成小的可交付单元，每完成一个就有正反馈，不会觉得遥遥无期。质量方面我不会因为赶工就跳过CR和自测，越忙越要守住底线，线上出了问题返工更耗时间。如果确实扛不住，我会主动跟leader沟通，说清楚当前的工作量和风险，争取资源支持或调整排期，而不是硬扛到最后交不出来。保持节奏感很重要，该休息就休息，疲劳状态下写的代码质量一定差。

## 5. 怎么看待工作和家庭的平衡？
说实话完全平衡很难，但可以做到动态调整。工作忙的时候比如大促或者重要项目上线，我会全力投入，家里人也理解。平时的话我会提高工作效率，尽量不把低效加班当常态，能在工作时间内高质量完成的就不拖到晚上。我觉得关键是提前沟通和规划，忙的时候跟家人说清楚这段时间的节奏，不忙的时候多陪陪家人。而且工作做好了心态好，回家状态也好；家庭稳定了，工作也更能全身心投入。两者不是对立的，是相互支撑的。

## 6. 怎么平衡架构和项目管理的时间
架构和项目管理不是割裂的，好的架构设计本身就是在为项目管理减负。
我的做法是把架构工作融入项目节奏里。需求评审阶段就把架构设计做了，不是单独找时间做架构，而是在方案设计环节一并考虑。技术方案评审就是架构决策的时机，不需要额外开"架构会"。
日常时间分配上，大概三七开。七成时间跟着项目节奏走，需求评审、方案设计、CR、上线保障这些。三成时间做架构演进，比如技术债治理、性能优化、系统重构，这些排进迭代计划里，跟业务需求一起排优先级，不是业余时间偷偷做。
关键是培养团队，把一部分项目管理的事情下放给骨干同学，比如让他们主导需求排期和进度跟踪，我就能腾出更多精力关注架构层面的事情。

## 7. 在工作中，怎么合理安排自己架构、开发、项目管理的时间。
我一般按项目阶段动态调整比例，不是固定分配。
项目前期，架构占大头，大概50%的精力在方案设计和技术评审上，30%在项目规划和排期，20%写核心模块的代码。这个阶段架构决策最重要，方向错了后面全白干。
项目中期，开发和项目管理为主。架构方案定了就不用天天想了，精力转到CR、解决技术难点、跟进进度和风险上。大概20%架构（处理设计变更）、40%开发（核心链路我自己写）、40%项目管理（站会、进度跟踪、协调上下游）。
项目后期，项目管理占大头，盯测试、盯上线、盯监控。架构基本不动了，开发收尾修bug。
日常没项目的时候，就是架构演进的好时机，做技术债治理、性能优化、工具建设。
核心原则是不要什么都自己干，架构和核心决策我把控，执行层面的开发和项目跟进尽量培养团队成员分担。
