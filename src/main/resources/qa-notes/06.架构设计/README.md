# 架构设计 Q&A

## 1. 停车场停车支付的架构设计
整体架构分三层：感知层、业务层、支付层。
感知层：入口车牌识别（OCR+摄像头），出口车牌识别，车位检测（地磁/超声波传感器）。数据通过MQTT/HTTP推送到业务层。
业务层核心域：
- 车辆管理域：车牌绑定、会员车辆、黑白名单
- 停车记录域：入场记录、出场记录、停车时长计算，状态机流转（入场→停车中→待支付→已支付→出场）
- 计费域：策略模式支持多种计费规则（按时计费、阶梯计费、封顶计费、免费时段），不同停车场不同规则，用规则引擎动态配置
- 车位域：实时车位统计，满位引导
支付层：
- 聚合支付，对接微信/支付宝/ETC/无感支付
- 预支付模式：用户扫码提前支付，出口直接抬杆，减少出口排队
- 无感支付：车牌绑定支付账户，出场自动扣款，走异步扣款+本地消息表保证一致性
- 支付超时处理：状态机管理，超时未支付回退到待支付状态
高并发场景（大型商场停车场）：
- 入场高峰用MQ削峰，车牌识别结果异步入库
- 计费结果缓存到Redis，出场时直接取不用重算
- 车位数用Redis原子操作DECR/INCR，不走DB
数据存储：热数据（当天停车记录）走MySQL，历史数据定期归档到HBase。车位实时状态走Redis。
容错：出口设备断网时本地缓存放行记录，网络恢复后补传。支付失败走人工通道兜底。

## 2. 福利社员工使用积分购买的交易的架构设计
核心是积分当货币用的交易系统，本质跟电商下单一样，只是支付介质从钱变成了积分。
业务域划分：
- 积分账户域：员工积分余额管理，积分发放（工龄/绩效/活动）、积分过期、积分流水。用账户+流水双表设计，余额变更必须有流水记录，方便对账。
- 商品域：福利商品管理，SKU、库存、上下架。库存用Redis预扣+DB最终扣减，防超卖。
- 交易域：下单、支付、退款。状态机流转：待支付→积分扣减中→已支付→待发货→已完成/已退款。
- 履约域：实物走物流，虚拟商品（卡券）直接发放到账。
关键设计点：
积分扣减的一致性：下单时先冻结积分（预扣），确认支付后真正扣减，取消订单解冻。类似TCC模式，Try冻结、Confirm扣减、Cancel解冻。积分账户用乐观锁update balance=balance-amount where balance>=amount防并发超扣。
防刷：同一员工同一商品限购，用Redis setnx做频率控制。热门商品用令牌桶限流。
积分过期处理：积分按批次管理，每批有过期时间，扣减时按FIFO先扣快过期的。定时任务扫描过期批次清零。
退款：积分原路退回，退回的积分保持原过期时间，不能变成永久积分。
数据存储：积分流水量大，热数据MySQL，历史流水归档到HBase。商品和订单走MySQL分库分表，按员工ID分片。
高可用：积分余额是核心资产，DB主从+半同步复制，不能丢数据。下单链路做熔断降级，积分服务挂了订单页展示"系统繁忙"而不是让用户下单失败后扣了积分。

## 3. 结合了AI大模型的晋升评审系统怎么设计
核心思路是AI做辅助评估+人工做最终决策，AI不能替代人做晋升决定，但能大幅提升评审效率和客观性。
业务域划分：
- 申请域：员工提交晋升申请，包含述职PPT、项目成果、绩效数据、代码贡献等材料。
- AI评估域：大模型对申请材料做多维度分析，输出评估报告。
- 评审域：评委基于AI报告+个人判断打分，支持多轮评审（初审→复审→终审）。
- 决策域：汇总评分，按规则生成晋升结果，支持人工干预调整。
AI大模型怎么用：
述职材料分析：把PPT/文档丢给大模型做结构化提取，自动识别项目规模、技术难度、业务价值、个人贡献度。用RAG检索该员工历史绩效、代码提交、CR记录、oncall表现等数据作为上下文，让模型给出更准确的评估。
多维度打分：模型按预设的评估维度（技术深度、业务理解、团队影响力、创新性）分别打分并给出理由。Prompt里注入公司的晋升标准作为评分依据。
横向对比：同批次申请人的材料做横向对比分析，识别差异化亮点和共性不足，辅助评委做相对排名。
评委辅助：评审时AI实时生成追问建议，比如"该候选人提到的XX方案缺少容灾设计，建议追问"。
关键设计点：
AI输出可解释性：每个评分必须附带理由和依据材料原文引用，评委能追溯AI为什么这么打分。不能是黑盒。
防偏见：多个模型交叉评估取平均，避免单一模型的偏见。定期用历史晋升数据校准模型输出。
数据安全：员工绩效、薪资等敏感数据不能外传，大模型必须私有化部署或用企业专属API，不走公网。材料存储加密，访问权限严格控制。
状态机：申请提交→AI评估中→AI评估完成→初审中→初审完成→复审中→复审完成→结果公示→已归档。每个状态流转有审计日志。
技术选型：大模型推理走异步，申请提交后发MQ触发AI评估，评估完回调更新状态。评估结果缓存到Redis，评委打开时秒级加载。材料文件存OSS，大文件不走DB。

## 4. 电商类目属性系统怎么设计
类目属性是电商的基础设施，商品、搜索、推荐、营销都依赖它，设计好坏直接影响整个平台的扩展性。
核心模型设计：
类目树：树形结构，前台类目（用户看到的导航）和后台类目（运营管理用的）分离。后台类目稳定，前台类目灵活调整映射关系，大促时可以临时挂新的前台类目而不动后台数据。类目表用parent_id+path（如/1/23/456/）存储，path字段方便用like查所有子类目，避免递归查询。
属性模型：三层结构——属性组、属性、属性值。比如"手机"类目下有属性组"基本参数"，包含属性"屏幕尺寸"，属性值"6.1英寸/6.7英寸"。
属性分类型：
- 关键属性：决定SPU的，比如品牌+型号，不同关键属性值就是不同SPU
- 销售属性：决定SKU的，比如颜色+内存，组合出具体SKU
- 非关键属性：描述性的，比如重量、材质，不影响SPU/SKU划分
属性与类目绑定：类目-属性关联表，不同类目挂不同属性集。叶子类目继承父类目的属性，也可以覆盖。这样"手机"和"电脑"共享"品牌"属性，但各自有独有属性。
属性值类型支持：枚举型（颜色：红/蓝/黑）、输入型（重量：用户填）、范围型（价格区间）。枚举值统一管理，防止同一个颜色叫"黑色""纯黑""曜石黑"。
存储设计：类目和属性定义走MySQL，结构化查询。商品的属性值用EAV模型（Entity-Attribute-Value）或JSON字段存储，灵活应对不同类目属性差异大的问题。搜索场景把属性值同步到ES，支持属性筛选和聚合。
缓存策略：类目树变更不频繁，全量缓存到Redis，用Hash存每个类目的属性列表。变更时通过MQ广播失效通知。前端类目导航用本地缓存+短TTL。
扩展性：新增类目只需要配置属性绑定关系，不用改代码。属性模板机制，相似类目可以复用同一套属性模板，减少重复配置。支持多语言属性名和属性值，国际化场景用属性ID关联多语言翻译表。

## 5. 权限中台怎么设计
权限中台核心是把权限能力从各业务系统抽出来统一管理，所有系统接入一套权限服务，避免每个系统各搞一套。
权限模型设计：
RBAC为基础，扩展ABAC。用户→角色→权限三层，角色是权限的集合，用户通过角色获得权限。在此基础上加属性维度（部门、职级、数据范围），支持更细粒度的控制。
功能权限：菜单权限（能不能看到这个页面）、按钮权限（能不能点这个操作）、接口权限（能不能调这个API）。三层对应前端渲染控制+后端拦截校验，不能只靠前端隐藏按钮。
数据权限：控制能看到哪些数据。按组织架构维度，比如区域经理只能看自己区域的订单。实现方式是SQL拦截，在MyBatis拦截器里自动拼接数据范围条件，业务代码无感知。
核心域划分：
- 用户域：用户、组织架构、用户组
- 角色域：角色定义、角色继承（角色树，子角色继承父角色权限）
- 资源域：菜单、按钮、API接口、数据资源统一抽象为"资源"
- 策略域：权限策略引擎，支持基于属性的动态规则，比如"工作日9-18点才能审批"
多租户支持：每个接入系统是一个租户，权限数据按租户隔离。租户可以自定义角色和权限，中台提供超级管理员跨租户管理。
鉴权流程：请求到网关→网关带token调权限中台鉴权接口→中台查Redis缓存的用户权限数据→返回是否放行。热路径全走缓存，权限变更时MQ广播失效。
缓存设计：用户的角色和权限列表缓存到Redis，key是userId，value是权限码集合。角色变更时批量失效该角色下所有用户的缓存。用布隆过滤器快速判断"肯定没权限"的请求，减少Redis查询。
高可用：权限中台挂了所有系统都不能用，所以要做降级。网关本地缓存一份权限快照，中台不可用时用本地缓存兜底，允许短暂的权限数据不一致。
审计：所有权限变更操作记审计日志，谁在什么时间给谁加了什么权限，不可篡改，存到独立的审计库。

## 6. 权限中台最大的挑战是什么
最大的挑战是通用性和灵活性的平衡。
每个业务系统的权限需求差异很大。电商要控制商品数据按品类隔离，CRM要按客户归属隔离，OA要按组织架构层级隔离。中台要用一套模型覆盖所有场景，抽象太高业务用不了，太贴合某个业务其他系统又接不进来。
具体几个难点：
数据权限是最难的。功能权限好抽象，就是"能不能操作"，但数据权限是"能看到哪些数据"，每个业务的数据隔离维度完全不同。有的按部门，有的按区域，有的按商家，有的是多维度交叉。中台要提供灵活的数据范围表达能力，又不能让配置复杂到没人会用。
性能压力。权限中台是所有请求的必经之路，每个API调用都要鉴权。QPS是所有接入系统的总和，稍微慢一点就拖垮全链路。缓存策略要极致优化，但权限变更又要及时生效，缓存一致性很难做。
权限爆炸。角色多了之后权限组合指数级增长，运营配错权限的风险很高。比如给一个角色加了个高危权限没人发现，就是安全事故。需要权限审计、风险检测、变更审批流，但这些又增加了使用成本。
存量系统迁移。各业务系统已经有自己的权限逻辑，迁移到中台要兼容历史数据和历史逻辑，不能让业务停服改造。需要提供适配层和灰度切换能力，这个过渡期很痛苦。

## 7. 大额支付的架构怎么设计
大额支付核心是安全第一、资金不能出错，跟普通支付比多了风控、审批、合规三道关。
安全层设计：
- 多因素认证：大额交易必须密码+短信+人脸识别三重验证，金额越大验证越严格，阶梯式策略。比如1万以下密码就行，1-10万加短信，10万以上加人脸。
- 设备指纹：识别异常设备，新设备首次大额支付强制人工审核。
- 交易限额：单笔限额、日累计限额、月累计限额，不同用户等级不同限额。
风控层设计：
- 实时风控：支付请求先过风控引擎，规则引擎+AI模型双重判断。规则引擎做硬规则（黑名单、异地登录、短时间多笔大额），AI模型做行为分析（跟用户历史交易模式偏差大的标记可疑）。
- 风控结果三种：放行、拦截、人工审核。大额支付的人工审核比例要高于普通支付。
- 用Flink做实时流计算，毫秒级出风控结果，不能让用户等太久。
资金安全设计：
- 幂等：用唯一支付单号做幂等键，防止重复扣款。支付请求先写本地消息表，再调银行接口，保证原子性。
- 对账：T+1全量对账，大额交易额外做T+0实时对账，发现差异秒级告警。
- 资金流水：每笔大额交易双向记账，借贷必须平衡。流水不可篡改，写入后只能追加不能修改。
- 分布式事务：跨系统扣款用TCC模式，Try冻结资金→Confirm扣减→Cancel解冻。超时未确认自动回滚。
审批流设计：
- 超过一定金额走审批，审批链路按金额阶梯配置。10万以上主管审批，100万以上总监审批。
- 审批走异步，不阻塞支付链路。用户提交后状态变为"审批中"，审批通过后自动触发扣款。
合规设计：
- 大额交易报告：按监管要求，单笔超过5万的现金交易、20万以上的转账自动上报反洗钱系统。
- 交易记录保留至少5年，审计可追溯。
- PCI DSS合规，敏感数据加密存储和传输。
高可用：支付链路做同城双活，主链路挂了秒级切备链路。资金类操作宁可拒绝也不能重复扣款，熔断策略偏保守。

## 8. 对接银行或支付清算系统的步骤
对接银行/清算系统是支付系统的核心链路，分几个阶段：
前期准备：
- 资质申请：拿到支付牌照或跟有牌照的机构合作，签银行合作协议，开通商户号。每家银行的接入流程不同，周期几周到几个月不等。
- 专线网络：跟银行之间走专线或VPN，不走公网。银行会分配IP白名单，只有白名单内的IP才能调接口。
- 证书交换：双方交换加密证书，报文用RSA/SM2签名+AES/SM4加密，防篡改防窃听。国内银行很多要求用国密算法。
技术对接：
- 报文协议：银行接口一般是XML或JSON over HTTPS，老系统可能是Socket+定长报文。每家银行报文格式不同，要做适配层统一封装，内部用标准模型，对外按银行要求转换。
- 核心接口：代扣（从用户银行卡扣款）、代付（打款到用户银行卡）、余额查询、交易状态查询、退款。每个接口都要做幂等，用商户订单号做唯一键。
- 异步通知：银行处理完会回调通知结果，但通知可能丢失或延迟，必须做主动查询兜底。定时任务轮询未终态的交易，调银行查询接口确认最终状态。
清算对账：
- T+1对账：每天银行会给一个对账文件（通常是SFTP下载），包含前一天所有交易明细。用对账系统逐笔比对，分三种结果：一致、我方有银行没有（长款）、银行有我方没有（短款）。
- 差错处理：长款可能是通知丢了要补单，短款可能是我方漏记要补录。差错走人工审核流程。
- 资金清算：对账无误后触发清算，计算手续费、分润，生成结算单，按约定周期（T+1/T+3）把资金从银行备付金账户划转到商户账户。
容错设计：
- 银行接口超时不代表失败，必须查询确认，绝对不能直接当失败退款，可能造成资金损失。
- 银行系统有维护窗口（通常凌晨），这个时段的交易要排队等恢复后重试。
- 多银行通道做路由，一家银行挂了自动切到备用通道，保证支付成功率。

## 9. 淘宝上面买了个东西，交易系统会创建一笔订单，向账户系统发一笔扣款请求，但是没有响应，交易订单就会处于处理中的状态，怎么保证最终一致性XA和TCC不能解决吗
XA和TCC理论上能解决，但在这个场景下不是最优选择。
XA的问题：XA是两阶段提交，要求交易系统和账户系统都支持XA协议，两边锁资源等协调者发commit。账户系统没响应的话，XA协调者也卡住了，所有参与者的资源都被锁着，吞吐量直接崩。而且XA跨系统依赖分布式事务管理器，淘宝这种量级根本扛不住同步阻塞。
TCC的问题：TCC能用，Try阶段冻结余额，Confirm扣款，Cancel解冻。但问题是账户系统没响应，你连Try有没有成功都不知道。TCC要求Try阶段必须有明确结果，超时无响应时Cancel可能空回滚（账户系统其实没收到Try），或者Try其实成功了但网络丢了响应，Cancel又会误回滚。虽然TCC有空回滚和悬挂的处理机制，但增加了很大的复杂度。
实际淘宝用的是本地消息表+最终一致性：
交易系统下单时，在同一个本地事务里写订单记录+写一条扣款消息到本地消息表，事务提交保证两者原子性。然后异步发MQ通知账户系统扣款。
账户系统消费消息执行扣款，成功后回调交易系统更新订单状态。如果账户系统没响应或消费失败，MQ自动重试。重试多次还失败进死信队列人工处理。
交易系统有定时任务扫描本地消息表，长时间未确认的消息重新投递。同时定时查询账户系统确认扣款结果，双保险。
账户系统接口做幂等，同一笔订单号重复扣款直接返回成功，不会多扣。
这样订单处于"处理中"是正常的中间态，最终要么扣款成功订单变"已支付"，要么超时未扣款订单自动关闭释放。不追求实时强一致，接受秒级的最终一致，但保证不丢不重。

## 10. 淘宝上面买了个东西，交易系统会创建一笔订单，向账户系统发一笔扣款请求，但是没有响应，交易订单就会处于处理中的状态，怎么保证最终一致性如果发起了重试，怎么避免重复入账
核心就是幂等设计，让同一笔扣款请求不管来几次，只执行一次。
具体实现：
唯一业务流水号：交易系统发扣款请求时带一个全局唯一的交易流水号（比如订单号+业务类型），账户系统用这个流水号做幂等键。
账户系统的处理逻辑：收到扣款请求后，先拿流水号去幂等表（唯一索引）做insert，插入成功说明是第一次，执行扣款；插入失败说明已经处理过，直接查上次的结果返回。幂等判断和扣款在同一个本地事务里，保证原子性。
```sql
-- 幂等表，流水号唯一索引
INSERT INTO payment_idempotent(biz_no, status, result) VALUES('ORDER_PAY_123456', 'PROCESSING', null);
-- 插入成功，执行扣款
UPDATE account SET balance = balance - amount WHERE user_id = ? AND balance >= amount;
-- 扣款成功，更新幂等表
UPDATE payment_idempotent SET status = 'SUCCESS', result = '扣款成功' WHERE biz_no = 'ORDER_PAY_123456';
```
重试来了的时候，insert幂等表会因为唯一索引冲突失败，直接查幂等表返回上次结果，不会重复扣款。
还有一种情况要注意：第一次请求到了账户系统，幂等记录插入了但扣款还没执行完，这时候重试来了。这时候幂等表已经有记录了，但状态是PROCESSING。处理方式是重试请求发现状态是PROCESSING就等一下再查，或者直接返回"处理中"让交易系统稍后再查。
Redis也可以做前置快速判断，用setnx流水号做快速去重，减少DB压力。但最终的幂等保障必须靠DB的唯一索引，Redis只是加速层。

## 11. 如果去查一下订单，再发起重试，这个有时间差的时候，怎么避免重复入账
这个时间差的问题本质是并发控制，查和重试之间状态可能变了。
关键在于幂等判断不是在调用方做的，是在账户系统侧做的。不管交易系统查没查、什么时候发重试，账户系统收到请求后自己用流水号做幂等判断，跟调用方的时序无关。
具体场景：交易系统查了一下发现订单还是"处理中"，于是发起重试。这时候可能有两种情况：
第一种，第一次请求其实已经到了账户系统正在处理。重试请求也到了，两个请求并发打到账户系统。这时候靠DB唯一索引保证：两个请求同时insert幂等表，只有一个能成功（行锁+唯一索引），另一个插入失败直接返回，不会重复扣款。
第二种，第一次请求已经扣款成功了但响应丢了。重试请求来了，insert幂等表失败，查到上次结果是SUCCESS，直接返回成功，也不会重复扣。
所以不管时间差多大、并发多少次，保障链路是：
```
请求到达 → insert幂等表(唯一索引) 
  → 插入成功 → 扣款 → 更新幂等状态
  → 插入失败(重复) → 查幂等表返回上次结果
```
这个判断是原子的，在DB层面用唯一索引+事务保证，不存在时间差问题。调用方随便重试，账户系统自己兜住。
如果要更严格，扣款操作本身也加乐观锁：update account set balance=balance-amount where user_id=? and balance>=amount and version=?，防止并发扣款。

## 12. 如果结合阿里的TCC分布式事务，那扣款的思路应该怎么设计？
TCC模式下扣款要拆成三个阶段，核心是引入"冻结金额"的概念。
账户表设计：balance（可用余额）+ frozen_amount（冻结金额）。
Try阶段（冻结）：不真正扣钱，只是把钱从可用余额转到冻结金额。
```sql
UPDATE account SET balance = balance - 100, frozen_amount = frozen_amount + 100 
WHERE user_id = ? AND balance >= 100;
```
这一步成功说明用户钱够，资源已预留。失败就直接返回余额不足，不用走后面的流程。
Confirm阶段（确认扣款）：订单确认后，把冻结金额真正扣掉。
```sql
UPDATE account SET frozen_amount = frozen_amount - 100 
WHERE user_id = ? AND frozen_amount >= 100;
```
钱已经从可用余额扣了，这步只是清掉冻结记录，表示这笔交易完成了。
Cancel阶段（回滚）：订单取消或超时，把冻结的钱还回可用余额。
```sql
UPDATE account SET balance = balance + 100, frozen_amount = frozen_amount - 100 
WHERE user_id = ? AND frozen_amount >= 100;
```
三个异常场景必须处理：
空回滚：Cancel来了但Try没执行过（网络丢了），Cancel要能识别出来直接返回成功，不能把没冻结的钱加回去。靠幂等表判断，没有Try记录就是空回滚。
悬挂：Cancel先到了，Try后到了。Cancel已经标记这笔交易结束了，后来的Try不能再冻结。Try执行前先查幂等表，发现已经有Cancel记录就拒绝执行。
幂等：三个阶段都要做幂等，同一笔流水号重复调用直接返回上次结果。
Seata的TCC模式就是这个思路，框架帮你管理Try/Confirm/Cancel的调度和超时回滚，业务方只需要实现三个接口。交易系统作为TCC发起方，账户系统作为参与方提供这三个接口。

## 13. 预留资金这个操作，在数据库的维度上发生了什么变动
预留资金在DB层面就是一条update，把钱从一个字段挪到另一个字段，总额不变。
具体变动：账户表有两个金额字段，balance（可用余额）和frozen_amount（冻结金额）。预留操作就是：
```sql
UPDATE account 
SET balance = balance - 100, 
    frozen_amount = frozen_amount + 100,
    version = version + 1
WHERE user_id = ? AND balance >= 100 AND version = ?;
```
执行前：balance=500, frozen_amount=0
执行后：balance=400, frozen_amount=100
用户总资产没变（500=400+100），但可用余额少了100，这100被"锁住"了，不能再用于其他交易。
同时要写一条冻结流水记录：
```sql
INSERT INTO account_transaction(biz_no, user_id, type, amount, before_balance, after_balance, status)
VALUES('ORDER_123', ?, 'FREEZE', 100, 500, 400, 'SUCCESS');
```
流水表记录了这笔冻结的来龙去脉，方便对账和审计。
where条件里的 balance >= 100 保证不会超扣，version乐观锁防并发。影响行数为0说明余额不足或被并发修改了，业务层重试或返回失败。

## 14. 事务悬挂是什么意思
事务悬挂就是Cancel比Try先执行了，导致Try后来到了却冻结了一笔永远不会被释放的资金。
场景是这样的：交易系统调账户系统的Try接口冻结资金，网络超时了，交易系统以为失败了就触发Cancel回滚。Cancel先到了账户系统，发现没有冻结记录（因为Try还没到），做了空回滚返回成功。
这时候网络恢复了，之前那个Try请求姗姗来迟到了账户系统，如果不做防护就会正常执行冻结操作。但Cancel已经执行完了，不会再来第二次Cancel，这笔冻结的钱就永远卡在frozen_amount里了，这就是悬挂。
解决方案：Cancel执行时如果发现没有Try记录（空回滚），要在幂等表里插入一条标记，记录这笔交易已经被Cancel了。后来Try到了先查幂等表，发现已经有Cancel记录就拒绝执行，直接返回失败。
```sql
-- Cancel时发现没有Try记录，插入防悬挂标记
INSERT INTO tcc_fence(biz_no, status) VALUES('ORDER_123', 'CANCELLED');

-- Try执行前先检查
SELECT status FROM tcc_fence WHERE biz_no = 'ORDER_123';
-- 如果查到CANCELLED，拒绝执行Try
```
Seata框架里的TCC Fence机制就是自动处理这个问题的，框架层面帮你做空回滚和防悬挂的判断。

## 15. TCC这种方案可以发生在高并发下同一个账户的不同扣款操作吗？
可以，TCC天然支持同一账户并发扣款，这也是它比XA好的地方。
核心在于Try阶段只冻结各自的金额，不互相阻塞。比如账户余额500，同时来了两笔扣款：
订单A的Try：冻结100，balance从500变400，frozen从0变100
订单B的Try：冻结200，balance从400变200，frozen从100变300
两笔Try操作各自独立，靠乐观锁（version）或者balance>=amount的条件保证不超扣。只要余额够，多笔并发冻结都能成功。
后续Confirm/Cancel也是各自独立的，订单A确认扣100，订单B取消退200，互不影响。因为每笔操作只操作自己冻结的那部分金额，通过biz_no区分。
跟悲观锁方案的区别：如果用select for update锁住整行，同一账户的扣款就变成串行了，一个没处理完其他的都排队等。TCC的Try只是做金额加减，行锁持有时间极短（一条update的时间），并发度高很多。
但有个边界情况：余额只剩100，两笔Try同时来各冻结100，只有一个能成功。这靠where balance >= amount保证，失败的那笔直接返回余额不足，业务层处理就行。
所以TCC在高并发同账户场景下是可行的，比2PC和悲观锁方案并发性能好得多。

## 16. 这种TCC方案在高并发下同一个账户的不同扣款操作，第一个请求过来，第二阶段还没提交。第二个请求刚来，应该是拒绝，还是执行第一阶段？
应该执行第一阶段，不应该拒绝。
这正是TCC设计的精髓所在。第一个请求Try冻结了100，Confirm还没来，这时候balance已经减了100，frozen加了100。第二个请求来了执行Try，看到的balance是扣掉冻结后的可用余额，只要余额够就继续冻结，不需要等第一个请求的Confirm。
举个例子：余额500
- 请求A Try冻结100：balance=400, frozen=100 ✓
- 请求B Try冻结200：balance=200, frozen=300 ✓（看到的可用余额是400，够扣）
- 请求A Confirm：frozen=200（减掉A的100）
- 请求B Cancel：balance=400, frozen=0（退回B的200）
每笔交易只关心可用余额够不够冻结自己的金额，不关心别人处于什么阶段。这就是TCC比2PC好的地方——Try阶段已经把资源隔离了，后续阶段互不干扰。
只有一种情况会拒绝：可用余额不够了。比如余额500，A冻结了400，B来冻结200发现balance只剩100不够，where balance >= 200条件不满足，update影响行数为0，返回余额不足。
所以TCC的并发控制不是靠"等前一个完成"，而是靠"冻结机制把资源预留好"，各自走各自的流程。这也是为什么TCC比悲观锁并发度高——不串行化，只要资源够就并行执行。

## 17. 商品库存先插入一条变更记录，先select..for update,再commit。这种操作怎么优化才能让其性能更高？
select for update是悲观锁，同一个SKU的扣减全部串行化，高并发下性能很差。几个优化方向：
第一，改乐观锁，去掉select for update：
```sql
-- 直接用原子update，不用先查再改
UPDATE stock SET quantity = quantity - 1 WHERE sku_id = ? AND quantity >= 1;
```
影响行数为1就是成功，为0就是库存不足。一条SQL搞定，不需要先select再update，行锁持有时间从整个事务缩短到一条update的执行时间。
第二，库存拆桶分散热点：
一个SKU的1000件库存拆成10个子记录，每个100件。扣减时随机选一个桶扣，把单行热点锁分散成10行，并发度提升10倍。某个桶扣完了再去其他桶借调。
第三，Redis预扣+异步落库：
库存预加载到Redis，用DECR原子扣减，扣成功的请求再通过MQ异步写DB。Redis单key十万QPS，远超DB行锁的并发能力。DB只做最终持久化，不在热路径上。
第四，变更记录异步写：
库存变更流水不要跟扣减在同一个事务里，扣减成功后发MQ异步写流水表。减少事务内的操作，缩短锁持有时间。
第五，合并扣减：
应用层攒一批请求合并成一次DB操作，比如100个扣1合并成扣100，减少DB交互次数。
优先级：先改乐观锁（最简单收益最大），再上Redis预扣（解决热点），最后考虑拆桶（极端场景）。

## 18. 如果有高并发库存超卖情况，使用Redis分布式锁的问题
用Redis分布式锁防超卖能解决正确性问题，但在高并发下有几个明显的问题：
性能瓶颈：分布式锁把并发扣减变成串行，同一个SKU同一时刻只有一个请求能扣库存，其他全部排队等锁。秒杀场景几万QPS打到一个SKU上，吞吐量直接降到锁的获取释放速度，可能只有几百TPS。
锁过期问题：业务还没执行完锁就过期了，另一个请求拿到锁也去扣库存，两个请求同时操作就可能超卖。Redisson的看门狗能续期，但如果GC暂停或网络分区，看门狗也续不了。
主从切换丢锁：Redis主从异步复制，Master加锁成功还没同步到Slave就挂了，Slave升主后锁丢失，两个客户端同时持有锁，并发扣减导致超卖。
锁粒度问题：如果锁粒度是SKU级别，同一个SKU的所有操作都串行化，包括查询库存这种读操作也被阻塞。锁粒度太粗影响性能，太细管理复杂。
更好的方案：
不用分布式锁，用Redis DECR原子操作直接扣减：
```bash
# 原子扣减，返回值<0说明库存不足，加回去
DECR stock:{sku_id}
```
DECR本身是原子的，不需要加锁，单key十万QPS，比分布式锁快两个数量级。扣减成功的请求再异步落DB，DB层用乐观锁update where quantity >= amount兜底防超卖。
所以高并发防超卖的正确姿势是Redis原子操作+DB乐观锁双保险，而不是分布式锁。

## 19. 房产商品的拍卖竞价，对高可用、高并发怎么设计
房产拍卖竞价的特点是：单品高价值、竞价频繁但总QPS不算极高、对数据正确性要求极高（出价不能丢、不能乱序）、有倒计时延时机制。
核心架构设计：
竞价引擎：
- 当前最高价用Redis存，ZSet结构，score是出价金额，member是用户ID+时间戳。每次出价先跟Redis里的最高价比较，高于当前价才接受。
- 出价操作用Lua脚本保证原子性：比较当前最高价→写入新出价→更新最高价，一个Lua脚本搞定，不需要分布式锁。
- 出价成功后发MQ异步落库，DB记录完整的竞价流水，Redis只保证实时竞价的正确性。
倒计时延时机制：
- 拍卖快结束时有人出价，自动延长倒计时（比如最后3分钟内出价延长5分钟）。用Redis的key过期+延时队列实现，每次出价重置倒计时。
- 结束时间用分布式定时任务（XXL-JOB）触发结拍，结拍时锁定竞价引擎不再接受出价。
高可用设计：
- 竞价数据Redis做主从+Sentinel，主挂了秒级切换。同时每笔出价都异步写DB，Redis丢了可以从DB恢复。
- 竞价服务多实例部署，无状态，网关层负载均衡。
- 结拍是关键操作，用分布式锁保证只有一个节点执行结拍逻辑，防止重复结拍。
高并发设计：
- 热门房产竞价高峰，出价请求先进MQ排队，消费者串行处理保证出价顺序。用户端乐观展示"出价提交中"，异步通知结果。
- 竞价详情页（当前价、出价记录）走CDN+本地缓存，WebSocket推送价格变更，不用轮询。
- 出价频率限制，同一用户1秒内只能出价一次，用Redis滑动窗口限流。
数据一致性：
- 出价金额必须大于当前最高价+最小加价幅度，Lua脚本原子校验。
- 结拍后冻结竞拍保证金，走TCC模式：Try冻结保证金→Confirm确认成交→Cancel退还保证金。
- 竞价流水不可篡改，写入后只追加不修改，用于争议仲裁和审计。

## 20. 很多人在最后一分钟出了不同的价格，拍卖时间截止了，怎么确定到底是哪个人拍卖成功？
核心靠两个机制：服务端时间戳定序 + 结拍时刻原子快照。
每笔出价在服务端打时间戳，不信任客户端时间。出价记录包含：用户ID、出价金额、服务端接收时间戳（精确到毫秒）。所有出价按服务端时间戳排序，不是按客户端发送时间。
结拍时刻的处理：
定时任务触发结拍时，用Lua脚本原子操作：锁定竞价引擎（设置一个flag拒绝新出价）→ 读取当前最高价和出价人 → 返回结果。这三步在一个Lua脚本里，Redis单线程保证原子性，不会出现结拍瞬间又插入一笔出价的情况。
同一秒多人出价的判定规则：
- 金额不同：最高价者胜出，没有争议
- 金额相同：按服务端时间戳，先到的胜出（先出价优先）
- 极端情况时间戳也一样（毫秒级并发）：在时间戳后面拼一个自增序列号，保证全局唯一排序
延时机制兜底：最后N分钟内有出价自动延长，这样"最后一秒抢拍"的场景会被延时消化掉，给其他人反应时间，减少争议。
结拍后的确认流程：结拍结果先写入DB，状态标记为"待确认"。系统自动校验竞价流水，确认最高价出价记录的时间戳确实在截止时间之前。校验通过才变为"已成交"，校验失败走人工审核。
所有竞价流水和结拍快照都保留，不可篡改，作为法律凭证。房产拍卖涉及大额资金，必须有完整的审计链路。

## 21. 如果很多人在最后一分钟出了相同的价格，拍卖时间截止了，怎么确定到底是哪个人先拍卖成功？
同价的情况下，谁先出价谁赢，靠服务端的全局有序序列号来判定。
技术实现：每笔出价到达服务端时，分配一个全局递增的序列号。不能只靠时间戳，因为毫秒级并发下时间戳可能一样。
序列号生成方案：
- Redis INCR生成全局自增ID，每笔出价一个，原子操作保证不重复不乱序
- 或者用雪花算法，时间戳+机器ID+序列号，同一毫秒内序列号递增
出价记录结构：用户ID + 金额 + 服务端时间戳 + 全局序列号。
判定规则：
1. 先比金额，高的赢
2. 金额相同比序列号，小的赢（序列号小说明先到达服务端）
在Redis里用ZSet存储，score的设计是关键：把金额和序列号编码成一个score值。比如score = 金额 * 10000000000 + (MAX_SEQ - 序列号)，这样ZREVRANGE取最大score的就是金额最高且同价时最先出价的。
```bash
# score编码：金额放高位，序列号取反放低位
# 金额100万，序列号1001 → score = 1000000 * 10000000000 + (9999999999 - 1001)
ZADD auction:{item_id} <score> <user_id>
# 取最高出价者
ZREVRANGE auction:{item_id} 0 0 WITHSCORES
```
这样一条ZREVRANGE命令就能拿到最终赢家，不需要额外排序逻辑。同价时序列号小的score反而大（因为取反了），自然排在前面。
整个判定过程是确定性的，任何时候查结果都一样，不存在争议。

## 22. 在请求量很大的情况下，怎么让拍卖失败的人知道自己没有拍成功？
分两个阶段通知：竞价过程中的实时反馈 + 结拍后的最终通知。
竞价过程中：
WebSocket长连接推送，用户进入拍卖页面就建立WebSocket连接。每次有人出价成功，服务端通过MQ广播给所有连接该拍卖场的WebSocket网关，推送最新价格。其他用户看到价格被刷新就知道自己的出价已经不是最高了。
出价时同步返回结果：用户出价后接口直接返回"出价成功/已被超越"。如果出价低于当前最高价，Lua脚本里直接拒绝并返回当前最高价，用户立刻知道没拍到。
结拍后：
结拍时确定赢家后，需要通知所有参与者。赢家和失败者走不同的通知链路：
赢家：站内信+短信+APP推送三路并发，确保必达。同时触发保证金冻结和后续交易流程。
失败者：量大是关键问题。一场热门房产拍卖可能几千人参与，不能同步通知。用MQ异步处理，结拍后发一条消息，消费者批量查出所有参与者（除赢家外），分批发送通知。
```
结拍 → MQ消息 → 消费者查参与者列表 → 分批500人一组 → 并行发送站内信+推送
```
推送降级策略：APP推送失败的走短信兜底，短信也失败的记录到待通知表，用户下次打开APP时拉取未读通知。
保证金退还：失败者的保证金异步退还，退还成功后再发一条"保证金已退还"的通知。用定时任务扫描未退还的保证金做兜底。
用户主动查询：不管推送到没到，用户在"我的拍卖"页面随时能看到拍卖结果和保证金状态，这是最终兜底。

## 23. 出价排序和确认是在拍卖结果出来后吗
不是，排序是实时的，不是结拍后才排。
每笔出价进来的时候就实时排好序了。用Redis ZSet存储，ZADD的时候score已经编码了金额和序列号，插入的同时就自动排好了。任何时刻ZREVRANGE取第一个就是当前最高出价者。
整个过程是这样的：
竞价中（实时）：
- 用户A出价100万 → ZADD插入 → 当前最高价100万
- 用户B出价120万 → ZADD插入 → 当前最高价自动变成120万
- 用户C出价120万 → Lua脚本判断不高于当前最高价 → 直接拒绝，返回"出价需高于120万"
每笔出价的排序在写入那一刻就完成了，不需要事后排。
结拍时（确认）：
- 定时任务触发结拍 → Lua脚本原子操作：设置拍卖状态为"已结束" + 读取ZSet第一名 → 返回赢家
- 这一步只是"锁定结果"，不是"计算结果"，结果在竞价过程中已经实时算好了
结拍后（校验）：
- 从DB的竞价流水表里重新校验一遍，确认Redis里的结果跟DB流水一致
- 校验通过 → 状态变为"已成交"
- 校验不通过（极端情况Redis和DB不一致）→ 以DB流水为准，人工介入
所以排序是实时的，结拍只是"拍照定格"当前排名，确认是结拍后的校验环节。

## 24. 房产商品的拍卖竞价，离拍卖截止时间还有很久，很多人在在同一时间出了不同的价格，怎么确定到底是哪个人在这一轮环节中拍卖成功，其他拍卖失败的人怎么同时得到自己这一轮失败的结果
竞价过程中不存在"一轮"的概念，每笔出价都是独立实时处理的，出价的瞬间就知道结果了。
同一时刻多人出不同价格的处理：
所有出价请求打到服务端后，通过Lua脚本串行化处理（Redis单线程天然串行）。每笔出价进来：比较金额是否大于当前最高价 → 大于则更新最高价并返回"出价成功" → 不大于则直接返回"出价失败，当前最高价XXX万"。
假设同一秒A出150万、B出130万、C出160万，到达服务端的顺序是B→A→C：
- B出130万 → 高于当前价100万 → 成功，最高价变130万 → 返回B"出价成功"
- A出150万 → 高于130万 → 成功，最高价变150万 → 返回A"出价成功"，同时推送B"你已被超越，当前最高价150万"
- C出160万 → 高于150万 → 成功，最高价变160万 → 返回C"出价成功"，同时推送A和B"当前最高价160万"
失败者怎么实时知道：
同步返回：出价接口直接返回结果，低于当前最高价的请求秒级返回"出价失败"，用户立刻知道。
WebSocket推送：每次最高价变更，通过MQ广播到WebSocket网关，推送给所有在线用户。之前出价成功但被超越的人，收到推送就知道自己不再是最高了。
推送流程：
```
出价成功 → 发MQ(价格变更事件) → WebSocket网关消费 → 推送给该拍卖场所有连接用户
```
所以不需要等"一轮结束"再通知，每笔出价都是实时反馈：出价低的同步返回失败，被超越的异步推送通知。用户体验就是出价后立刻知道结果，页面上的最高价实时刷新。

## 25. 设计一个基于房产商品的秒杀系统，考虑高并发场景
房产秒杀跟普通商品秒杀的区别：单品高价值、库存极少（可能就一套）、需要资格审核、涉及大额资金冻结、法律合规要求高。
整体架构分层：
前端层：
- 秒杀页面静态化，HTML/JS/CSS推CDN，不打到后端。倒计时用客户端本地时间+服务端校准，不轮询。
- 秒杀按钮到点才激活，前端做防重复点击（点一次灰掉），但不能只靠前端控制。
网关层：
- 限流是第一道防线。Nginx层按IP限流，每个IP每秒最多1次请求。网关层按用户ID限流，同一用户重复请求直接拒绝。
- 资格校验前置：秒杀前用户必须完成实名认证+购房资格审核+缴纳保证金，没完成的请求在网关层直接拦掉，大幅减少到达后端的请求量。
服务层：
- Redis预扣库存：房产库存就1套，用DECR原子操作，第一个扣成功的拿到资格，后面的全部返回"已售罄"。一条命令搞定，不需要分布式锁。
- 扣减成功后发MQ，异步创建订单。用户端返回"抢购成功，请在30分钟内完成支付"。
- 订单服务消费MQ创建订单，同时冻结保证金（TCC模式）。
防刷策略：
- 同一用户同一房源只能抢一次，Redis setnx做去重。
- 风控引擎实时检测：同一设备多账号、异常IP段、机器人行为特征，命中的直接拦截。
- 验证码/滑块验证，增加机器人成本。
大额资金处理：
- 秒杀成功后进入支付环节，房产金额大，走银行大额支付通道。
- 支付超时（30分钟）自动释放库存，库存回到Redis，触发候补机制通知排队用户。
- 保证金走TCC：Try冻结→支付成功Confirm扣除→超时Cancel退还。
候补机制：
- 第一个人超时未支付，库存释放后通知候补队列里的下一个人。用Redis List做候补队列，LPUSH入队RPOP出队。
- 候补通知走短信+APP推送，给候补者限时支付窗口。
数据一致性：
- Redis库存和DB库存最终一致，Redis扣减成功后MQ异步同步DB。
- 对账定时任务比对Redis库存和DB订单数，不一致告警。
- 订单状态机：待支付→支付中→已支付→待签约→已完成/已超时释放。
高可用：
- Redis做主从+Sentinel，库存数据不能丢。
- 秒杀服务多实例无状态部署，任何一台挂了不影响。
- 降级方案：Redis挂了直接返回"系统繁忙"，不能降级到DB扣减，防止超卖。

## 26. 微博里面有很多好友，要查出所有好友里最新的一条信息。海量数据怎么解决
这就是经典的Feed流设计问题，核心有推模式、拉模式、推拉结合三种方案。
拉模式（读扩散）：
用户打开首页时，实时去查所有好友的最新一条微博。假设有500个好友，就要查500次取最新的，再合并排序。数据量大时读延迟高，但写入简单，发微博只写自己的表。适合好友少的场景。
推模式（写扩散）：
用户发微博时，把这条微博推送到所有粉丝的收件箱（inbox）。粉丝打开首页直接读自己的inbox就行，读很快。但大V发一条微博要写几千万个粉丝的inbox，写放大严重，存储成本也高。
推拉结合（实际方案）：
按用户分级处理。大V（粉丝超过一定阈值）用拉模式，普通用户用推模式。
具体实现：
- 普通用户发微博：推送到所有粉丝的inbox，用MQ异步写，不阻塞发布。inbox用Redis ZSet存，score是时间戳，只保留最近N条。
- 大V发微博：只写自己的发件箱（outbox），不推送。粉丝读首页时，先读自己的inbox（普通好友推过来的），再拉取关注的大V的outbox最新一条，合并排序。
- 拉取大V数据时用Redis缓存大V最新微博，不查DB。
海量数据存储：
- 热数据（最近7天）：Redis ZSet存inbox/outbox，按时间戳排序。
- 温数据（7-30天）：MySQL分库分表，按用户ID分片。
- 冷数据（30天以上）：归档到HBase，按rowkey（用户ID+时间戳反转）范围扫描。
查所有好友最新一条的优化：
- 不是每次都查所有好友，inbox里已经有推过来的数据了，只需要补拉大V的。
- 大V列表缓存在本地，大V最新微博缓存在Redis，一次MGET批量取，不用逐个查。
- 合并排序在应用层做，取TopN用小顶堆，O(NlogK)搞定。

## 27. 微博里面有很多好友，要查出所有好友里按时间排序的前N条信息或者全部信息。海量数据怎么解决
跟上一个问题的区别在于不是每人取一条，而是要取前N条或全部，数据量大很多。
取前N条（首页Feed流）：
还是推拉结合模式，但inbox里存的不是每个好友一条，而是按时间排序的混合流。
Redis ZSet做实时Feed：用户的inbox是一个ZSet，score是微博发布时间戳，member是微博ID。普通好友发微博推到inbox，大V的拉取合并。取前N条就是ZREVRANGE inbox:{userId} 0 N-1，O(logN+N)搞定。
翻页用游标分页：不用offset，用上一页最后一条的时间戳做游标，ZREVRANGEBYSCORE inbox:{userId} (lastScore +inf LIMIT 0 20。避免深度翻页性能问题。
inbox容量控制：只保留最近1000条，超过的裁剪掉。用户翻到1000条以后走DB查询。
取全部信息（历史浏览）：
这个不能走Redis了，数据量太大。分层查询：
第一层，Redis inbox取最近的热数据（最近7天或1000条）。
第二层，MySQL分库分表存中期数据。Feed流表按用户ID分库，每个用户的Feed按时间倒序存。查询时带上用户关注列表做IN查询+时间排序，但好友多了IN太大，所以还是用物化的Feed表（推模式写入的），直接按用户ID查自己的Feed表翻页。
第三层，HBase存历史全量。rowkey设计为userId_reverseTimestamp，这样同一用户的数据连续存储，按时间倒序扫描天然有序。Scan指定startRow和stopRow做范围查询，HBase对这种顺序扫描很快。
合并排序策略：
如果要跨层查（比如翻页翻到Redis和MySQL的交界处），用归并的思路。每层各取一批，应用层用小顶堆合并，按时间戳排序取TopN。维护每层的游标位置，下次翻页从各层游标继续取。
大V数据的处理：大V的微博单独存一份热点缓存，粉丝拉取时批量MGET所有关注大V的最新微博列表，跟inbox数据合并排序。大V微博量不大（一天几条），拉取成本低。
写入优化：推送到inbox用MQ异步批量写，RocketMQ按用户ID路由到同一Queue保证同一用户的Feed顺序。消费者批量ZADD减少Redis交互次数。

## 28. 是每个用户都建立一个队列吗
不是每个用户一个MQ队列，那几亿用户就几亿个队列，MQ扛不住。
MQ队列的设计是按分片，不是按用户。比如RocketMQ的Topic下设置256个Queue，用户ID hash取模路由到对应Queue。同一个用户的消息固定落到同一个Queue，保证该用户的Feed写入顺序。不同用户分散在不同Queue上并行消费。
但Redis的inbox确实是每个用户一个。每个用户一个ZSet，key是inbox:{userId}，这是合理的，因为Redis的key是轻量的，几亿个key没问题，单个ZSet只存最近1000条微博ID，内存可控。
区分一下两个层面：
MQ层面：少量Queue（几十到几百个），按用户ID hash分片，解决的是异步写入的吞吐和顺序问题。消费者从Queue取消息，解析出目标用户ID，写到对应用户的Redis inbox里。
Redis层面：每个用户一个ZSet（inbox），解决的是用户读Feed流时的快速查询。这是必须per-user的，因为每个人关注的人不同，Feed内容不同。
DB层面：Feed表按用户ID分库分表，每个用户的Feed数据在同一个分片里，查询时只扫一个分片。
所以总结就是：MQ按hash分片共享队列，Redis和DB按用户隔离数据。

## 29. 如果大V如果有1亿个粉丝，那么1亿个粉丝怎么解决。实际可行的解决方案
1亿粉丝的大V绝对不能用推模式，推一条微博要写1亿个inbox，几分钟都写不完，而且大V发微博频率高，写放大灾难级别。
实际方案就是大V只用拉模式，粉丝主动来拉。
大V的微博怎么存：
- 大V的outbox（发件箱）用Redis ZSet缓存最近100条微博ID，score是时间戳。全量数据在DB。
- 大V微博内容单独缓存，key是微博ID，value是微博内容，设置热点自动识别+本地缓存。一条热门微博可能被几千万人读，必须多级缓存：本地Caffeine → Redis → DB。
粉丝怎么读大V的微博：
- 粉丝打开首页时，先读自己的inbox（普通好友推过来的），再批量拉取关注的大V列表的最新微博。
- 关键优化：粉丝不需要拉所有关注的大V，只拉"上次读取之后有更新的大V"。用一个Redis Hash存每个大V的最新发布时间戳，粉丝端记住上次拉取时间，HMGET批量比对，只拉有新内容的大V。
- 一个用户关注的大V一般不超过几百个，HMGET几百个key毫秒级完成。
1亿粉丝同时来拉怎么扛：
- 大V微博内容是同一份数据，天然适合缓存。Redis单key十万QPS，加上本地缓存，一条热门微博的读取能力是无限的。
- 不是1亿人同时来，用户打开APP的时间是分散的。假设DAU 50%，高峰期集中在1小时内，QPS大概是5000万/3600≈1.4万QPS，Redis轻松扛住。
- 真正的热点微博（刚发的爆款），用本地缓存+Redis多副本打散，单条微博百万QPS没问题。
合并排序：
- 粉丝端拿到inbox数据+大V最新微博，应用层归并排序取TopN。数据量不大（inbox最近几十条+几个大V各一条），排序开销忽略不计。
这就是微博实际的做法：大V不推，粉丝来拉，靠缓存扛读。

## 30. 怎么实现多人实时在线编辑文档
核心要解决的问题是多人同时编辑同一个位置时的冲突合并，以及实时同步。
冲突解决算法（二选一）：
OT（Operational Transformation）：Google Docs用的方案。把每次编辑抽象成操作（insert/delete/retain），服务端收到多个客户端的操作后做变换（transform），调整操作的位置偏移量使得最终结果一致。比如A在位置5插入"hello"，B同时在位置3删除一个字符，服务端要把A的操作位置从5调整为4。需要中心服务器做变换，实现复杂但成熟。
CRDT（Conflict-free Replicated Data Type）：无冲突复制数据类型，每个字符有全局唯一ID和逻辑位置，多端并发编辑自动合并不需要中心协调。适合去中心化场景，但内存开销大。
实际架构设计：
通信层：WebSocket长连接，客户端每次编辑操作实时发到服务端，服务端广播给同文档的其他客户端。用Redis Pub/Sub做跨服务器的消息广播，同一文档的用户可能连在不同的WebSocket网关上。
协作服务：
- 每个文档一个协作会话，维护当前在线用户列表和光标位置。
- 操作日志顺序存储，每个操作带版本号（递增），客户端发送操作时带上自己基于的版本号，服务端做OT变换后分配新版本号广播。
- 用Redis存文档的实时状态和操作队列，保证同一文档的操作串行处理（单线程或分布式锁）。
存储层：
- 实时编辑状态存Redis，毫秒级读写。
- 操作日志持久化到DB，用于回放和版本历史。
- 定期做快照（snapshot），把当前文档状态完整保存，避免回放太多操作日志。
- 文档内容最终存储到对象存储或DB，用户关闭文档时触发保存。
高可用：
- WebSocket网关多实例，用户断连后自动重连到其他节点，重连时拉取断连期间的操作日志补齐。
- 协作服务无状态，文档状态在Redis里，任何实例都能处理。
- 离线编辑支持：客户端本地缓存操作，上线后批量同步，服务端做OT合并。

## 31. 实现可拖拽的晋升评审系统部门树怎么设计
核心是树形结构的存储设计要支持高效的拖拽移动（改变父节点和排序），同时保证查询性能。
数据库表设计：
用邻接表+路径+排序号三合一的方案：
```sql
CREATE TABLE department_tree (
    id BIGINT PRIMARY KEY,
    name VARCHAR(100),          -- 部门名称
    parent_id BIGINT,           -- 父节点ID，邻接表
    path VARCHAR(500),          -- 物化路径，如 /1/23/456/
    level INT,                  -- 层级深度
    sort_order INT,             -- 同级排序号
    version INT,                -- 乐观锁
    -- 晋升评审相关
    review_group_id BIGINT,     -- 关联的评审组
    reviewer_ids VARCHAR(500),  -- 评委ID列表
    status TINYINT              -- 启用/禁用
);
```
三种查询场景的支持：
- 查子节点：WHERE parent_id = ? ORDER BY sort_order（邻接表）
- 查所有后代：WHERE path LIKE '/1/23/%'（物化路径）
- 查祖先链路：直接拆path字符串，不用递归查询
拖拽操作的设计：
拖拽本质是两个操作：改变父节点 + 改变排序位置。
改变父节点（跨部门拖拽）：
```sql
-- 1. 更新被拖节点及所有后代的path
-- 旧path: /1/23/456/  新path: /1/78/456/
UPDATE department_tree 
SET path = REPLACE(path, '/1/23/', '/1/78/'),
    level = level + (新层级 - 旧层级),
    parent_id = 78  -- 只改被拖节点本身的parent_id
WHERE path LIKE '/1/23/456/%' OR id = 456;
```
改变排序（同级拖拽）：
排序号用间隔设计，初始值1000、2000、3000，拖拽到两个节点之间取中间值1500。间隔用完了批量重排。
```sql
-- 拖到A(1000)和B(2000)之间
UPDATE department_tree SET sort_order = 1500 WHERE id = ?;
-- 间隔太小时批量重排
UPDATE department_tree SET sort_order = ROW_NUMBER() * 1000 
WHERE parent_id = ? ORDER BY sort_order;
```
并发控制：拖拽操作用乐观锁（version字段），两个人同时拖同一个节点，后提交的检测到version变了就提示冲突重试。
缓存设计：整棵树缓存到Redis，用Hash存每个节点，key是dept_tree:{rootId}。拖拽操作成功后MQ广播失效缓存，前端重新拉取。树不大的话（几百个节点）直接全量缓存，前端本地做树的渲染和拖拽交互。

## 32. 结合了流程引擎的晋升评审系统怎么设计
晋升评审天然是多阶段审批流程，用流程引擎把评审环节从代码里抽出来，可配置可编排。
流程引擎选型：Flowable（Activiti的升级版），支持BPMN2.0规范，Java生态成熟。
晋升评审流程定义：
```
申请提交 → 直属主管初审 → 部门评委会评审 → HR复核 → 高管终审 → 结果公示
                ↓(驳回)              ↓(驳回)         ↓(驳回)
              退回修改            退回修改          退回修改
```
每个环节用Flowable的UserTask建模，环节之间用排他网关（ExclusiveGateway）做分支判断（通过/驳回/转审）。
核心设计：
动态审批链路：不同级别的晋升走不同流程。P6→P7走三级审批，P8→P9加高管终审。用流程变量（ProcessVariable）传入晋升级别，排他网关根据变量决定走哪条分支。流程定义存在Flowable的ACT_RE_PROCDEF表里，运营可以在后台可视化编辑流程图，不用改代码。
评委会并行评审：部门评委会环节用并行网关（ParallelGateway），多个评委同时收到待办，各自独立打分。用Flowable的多实例任务（Multi-Instance Task），配置评委列表，所有人完成或达到通过比例后自动流转到下一环节。
```xml
<!-- 多实例并行评审 -->
<userTask id="committeeReview" name="评委会评审">
  <multiInstanceLoopCharacteristics isSequential="false"
    completionCondition="${nrOfCompletedInstances/nrOfInstances >= 0.6}">
    <loopCardinality>${reviewerList.size()}</loopCardinality>
  </multiInstanceLoopCharacteristics>
</userTask>
```
超时处理：每个环节设置定时边界事件（Timer Boundary Event），超过3天未处理自动提醒，超过7天自动升级到上级处理。Flowable的Timer Job自动触发。
业务域设计：
- 申请域：晋升申请表，关联流程实例ID（processInstanceId），状态跟流程同步。
- 评审域：评审记录表，每个环节每个评委的打分、评语、结论。
- 流程域：Flowable自带的运行时表（ACT_RU_*）和历史表（ACT_HI_*），记录流程流转全过程。
跟业务系统的集成：
- Flowable通过TaskListener和ExecutionListener回调业务逻辑。比如评审通过后触发Listener发送通知、更新申请状态、冻结晋升名额。
- 待办任务通过Flowable的TaskService查询，推送到消息中心和APP。
- 流程状态变更发MQ通知业务系统，业务系统不直接操作Flowable的表。
数据存储：Flowable的表和业务表分库，Flowable用独立数据源。评审打分数据量不大走MySQL，历史评审数据归档到HBase做分析。

## 33. 假设一个电商系统，有一张用户表，平时写入工作不会很高，但是活动期间会突然激增，原有写逻辑就不太适合，在这种场景下怎么去做改造
核心思路是削峰+异步+弹性，平时走正常链路，活动期间自动切换到高并发模式。
MQ削峰是第一步：
活动期间用户注册/信息变更请求先写MQ，消费者按DB能承受的速率消费写入。用户端同步返回"处理中"，异步通知结果。平时直接同步写DB，通过配置开关切换。
```java
// 开关控制
if (isActivityMode) {
    mq.send("user_write_topic", userInfo); // 活动期间走MQ
    return "处理中";
} else {
    userDao.insert(userInfo); // 平时直接写
    return "成功";
}
```
批量合并写入：
消费者不是一条一条写DB，而是攒一批（比如500条）用batch insert一次写入，减少DB交互次数，吞吐量提升几十倍。
```sql
INSERT INTO user(id, name, phone...) VALUES 
(1,'a','xxx'), (2,'b','xxx'), ... (500,'z','xxx');
```
分库分表提升写入能力：
如果单表写入瓶颈明显，按用户ID hash分到多个库多个表，写入并行度从1变成N。活动前提前扩好分片，不要活动中临时扩。
弹性伸缩：
消费者服务部署在K8s上，配置HPA（Horizontal Pod Autoscaler），根据MQ消费lag自动扩容消费者实例。活动结束lag降下来自动缩容。
DB层面优化：
- 活动期间关闭非必要的索引更新，活动后rebuild
- 用户表拆分，核心字段（ID、手机号、状态）和扩展字段（地址、偏好）分表，减少单行写入的数据量
- 写入走主库，读走从库，读写分离减轻主库压力
兜底方案：
DB写入超过阈值触发限流，MQ堆积超过阈值告警扩容。极端情况下降级非核心写入（比如用户画像更新可以延后），保住核心注册链路。

## 34. 引入消息队列（如Kafka）缓冲高峰期写入请求，异步写入数据库，减轻数据库直接写入压力。在异步写的间隙，怎么保证数据的准确，页面上能查到最新的数据
核心问题是MQ异步写DB有延迟，用户提交后立刻查可能查到旧数据。解决思路是写入缓存做中间态，读的时候优先读缓存。
写入时双写Redis：
请求进来先写Redis缓存（同步），再发MQ异步写DB。用户查询时先查Redis，Redis有就返回最新数据，Redis没有再查DB。
```java
// 写入流程
redis.set("user:" + userId, userInfo, 10分钟); // 同步写缓存
mq.send("user_write_topic", userInfo);          // 异步写DB
return "成功";

// 查询流程
UserInfo user = redis.get("user:" + userId);    // 先查缓存
if (user == null) {
    user = userDao.selectById(userId);           // 缓存没有查DB
}
return user;
```
这样用户提交后立刻查，命中Redis返回最新数据，不用等DB写入完成。MQ消费者写完DB后缓存自然过期或主动删除，后续查询走DB。
数据准确性保证：
MQ消费失败怎么办：消费失败自动重试，重试多次进死信队列人工处理。Redis里的数据有TTL兜底，即使MQ丢了，缓存过期后查DB返回旧数据，用户能感知到异常再重新提交。
Redis和DB不一致：MQ消费成功写完DB后，主动删除Redis缓存（或更新），下次查询走DB拿到最新数据。用Canal监听binlog做兜底，DB变更后异步删缓存。
消息顺序：同一用户的多次修改要保证顺序，用userId做MQ的路由key，同一用户的消息落到同一个partition串行消费。
幂等：消费者用唯一请求ID做幂等，重复消费不会产生脏数据。
页面体验优化：前端提交后用本地状态直接更新页面展示（乐观更新），不等后端返回查询结果。如果后端最终失败了再回滚页面状态通知用户。

## 35. 新的项目比如物流，如果数据还比较海量，每天产生10亿轨迹流水数据，怎么做数据存储设计
每天10亿条轨迹数据，一年3600亿条，MySQL肯定扛不住，要按数据特征分层存储。
数据特征分析：轨迹数据是典型的时序数据，写多读少，写入量大但单条数据小（运单号+经纬度+时间戳+状态），查询模式主要是按运单号查轨迹和按时间范围查。
写入层设计：
Kafka做写入缓冲：轨迹上报先写Kafka，削峰填谷。10亿条/天≈1.2万条/秒，高峰期可能5-10万TPS，Kafka轻松扛住。
Flink实时消费：从Kafka消费做实时处理，包括轨迹去重（同一运单同一秒多次上报）、异常检测（轨迹跳变）、状态聚合（最新位置更新）。处理后分流写入不同存储。
存储层分级：
热数据（最近7天）：写入HBase。rowkey设计为运单号反转+时间戳，同一运单的轨迹连续存储，范围扫描高效。HBase天然支持海量写入，LSM树顺序写性能好。每天10亿条对HBase来说很轻松。
最新位置：Redis存每个运单的最新轨迹点，key是track:{waybillNo}，用户查物流时秒级返回当前位置。每次轨迹上报更新Redis。
温数据（7-90天）：HBase继续存，设置TTL自动过期或者用不同的列族区分冷热。
冷数据（90天以上）：归档到HDFS，用Parquet列式格式压缩存储，压缩率10:1以上。需要查的时候用Spark/Hive离线查询。
查询层设计：
按运单查轨迹：先查Redis拿最新位置，查全部轨迹走HBase按rowkey前缀扫描。
按区域/时间范围查（运营分析）：轨迹数据同步到ES建地理位置索引（geo_point），支持按区域圈选、按时间范围聚合。或者用ClickHouse做分析查询，列式存储聚合极快。
按运单号模糊搜索：ES倒排索引支持。
容量规划：
- 单条轨迹约200字节，10亿条/天≈200GB/天
- HBase存7天≈1.4TB，加副本3倍≈4.2TB
- HDFS存一年，压缩后≈7TB
- Redis只存最新位置，假设1亿活跃运单×200字节≈20GB
写入保障：Kafka做持久化，消费者挂了重启从offset继续消费不丢数据。HBase写入失败走重试+死信队列。定时对账Kafka消息数和HBase写入数，不一致告警。

## 36. 实际当中，如果遇到很多条件维度要查询，也建了很多索引，索引太多影响性能，有没有替代方案
多维度查询索引爆炸的问题，核心思路是把查询从MySQL迁移到更适合多条件检索的存储引擎。
ES/OpenSearch是首选方案：
把需要多维度查询的数据同步到ES，ES的倒排索引天然支持任意字段组合查询，不需要像MySQL那样为每种查询组合建联合索引。Canal监听binlog异步同步，MySQL只负责写和主键查询，ES负责复杂条件检索。
```
写入：应用 → MySQL（主存储）
同步：MySQL binlog → Canal → MQ → ES
查询：简单查询走MySQL，多条件查询走ES
```
ClickHouse适合分析型查询：
如果查询偏分析聚合（按时间段统计、分组汇总），ClickHouse列式存储比ES更快。每列独立存储和压缩，查询时只读需要的列，IO极少。
宽表+冗余字段：
如果不想引入新组件，把多表join的查询拍平成一张宽表，冗余存储常用查询字段。查询变成单表多条件，一个联合索引覆盖主要查询场景。代价是写入时要维护宽表数据一致性。
位图索引方案：
对于枚举值字段（状态、类型、等级），用Redis Bitmap或RoaringBitmap做交集运算。每个字段值一个bitmap，多条件查询就是多个bitmap做AND运算，毫秒级返回符合条件的ID列表，再回MySQL查详情。
MongoDB：
如果数据结构灵活、查询条件多变，MongoDB的文档模型+自动索引比MySQL灵活。支持任意字段建索引，复合索引的前缀匹配也比MySQL宽松。
选型建议：电商场景多条件商品搜索用ES，报表分析用ClickHouse，简单场景用宽表。核心原则是MySQL做写入和事务，复杂查询卸载到专门的查询引擎。

## 37. 怎么引入AI对商品发布更新等操作做自动化的审核和诊断
核心是AI做前置自动审核+诊断建议，人工做兜底复核，提升审核效率和商品质量。
审核链路设计：
```
商品提交 → 规则引擎(硬规则) → AI审核(软规则) → 自动通过/人工复核/自动驳回
```
第一层，规则引擎做硬规则校验：
必填字段缺失、价格为0或负数、类目不匹配、违禁词命中（黄赌毒、政治敏感）。这些确定性规则不需要AI，if-else或Drools规则引擎搞定，毫秒级返回。
第二层，AI大模型做软规则审核：
图片审核：调用视觉模型检测商品图片质量（模糊、水印、牛皮癣广告）、是否盗图（图片相似度检索）、是否含违规内容。用预训练的图像分类模型，推理走GPU集群。
文本审核：大模型分析标题和描述，检测虚假宣传（"全网最低"、"100%治愈"）、夸大描述、与类目不符的内容。用Prompt让模型按审核维度逐项打分并给出理由。
属性诊断：AI检查商品属性是否合理，比如手机重量填了50kg、衣服尺码缺失。用RAG检索同类目商品的属性分布，偏离太大的标记异常。
价格诊断：跟同类目同品牌的历史价格对比，价格异常偏高或偏低的标记可疑（可能是标错价或刷单）。
审核结果三档：
- 置信度高（>95%）且无风险：自动通过，不需要人工
- 置信度中等或有轻微风险：自动通过但标记待复核，人工抽检
- 置信度低或有明显风险：自动驳回，附带AI给出的驳回理由和修改建议
诊断建议输出：
AI不只是说"不通过"，还要告诉商家怎么改。比如"标题含夸大用语'全网最低'，建议修改为具体优惠信息"、"主图清晰度不足，建议上传800×800以上的图片"。用大模型生成自然语言的修改建议。
技术架构：
- AI推理走异步，商品提交后发MQ触发审核，审核完回调更新状态。不阻塞商品发布流程。
- 模型推理用GPU集群，图片和文本审核并行执行，总耗时控制在10秒内。
- 审核结果和AI打分存DB，用于模型效果评估和持续优化。
- 人工复核的结果反馈给模型做fine-tune，持续提升准确率。
高可用：AI服务挂了降级为纯规则引擎审核+全量人工复核，不能阻塞商品发布。

## 38. 设计一个云盘，有一个文件目录，有一个父级文件夹名字A，子级有个文件B.txt和它的大小，已经把这些信息都存到MYsql。现在想知道文件夹里所有子级文件的大小总和。
两种方案：实时计算和预计算。
表结构设计：
```sql
CREATE TABLE cloud_file (
    id BIGINT PRIMARY KEY,
    name VARCHAR(255),        -- 文件/文件夹名
    parent_id BIGINT,         -- 父级ID
    path VARCHAR(1000),       -- 物化路径 /1/23/456/
    type TINYINT,             -- 0文件夹 1文件
    file_size BIGINT,         -- 文件大小(字节)，文件夹为0
    total_size BIGINT,        -- 预计算：文件夹下所有文件大小总和
    version INT
);
```
方案一，实时计算（用物化路径）：
```sql
-- 查文件夹A下所有子级文件的大小总和
SELECT SUM(file_size) 
FROM cloud_file 
WHERE path LIKE '/1/23/%' AND type = 1;
```
path字段加索引，LIKE前缀匹配能走索引。文件少的时候够用，但文件夹下有几十万文件时查询慢。
方案二，预计算（推荐）：
文件夹维护一个total_size字段，文件增删改时沿着path向上逐级更新所有祖先文件夹的total_size。
```sql
-- 上传文件时，更新所有祖先文件夹的total_size
-- path为 /1/23/456/，祖先ID是1、23、456
UPDATE cloud_file 
SET total_size = total_size + #{fileSize}
WHERE id IN (1, 23, 456);
```
查询时直接读total_size字段，O(1)搞定，不用遍历子文件。
写入时多了一步更新祖先，但文件夹层级一般不超过10层，更新10行的开销很小。用MQ异步更新也行，允许短暂不一致。
删除和移动文件时要反向操作：旧路径的祖先减size，新路径的祖先加size。
并发控制：用乐观锁update total_size = total_size + ? where id = ? and version = ?，防止并发上传导致size不准。
实际云盘产品（百度网盘、阿里云盘）都用预计算方案，查询是高频操作，写入相对低频，用写入时的少量开销换查询时的极致性能。

## 39. 设计一个云盘，有一个文件目录，有一个父级文件夹名字A，子级有个文件B.txt和它的大小，子级里还有子级(递归)，已经把这些信息都存到MYsql。现在想知道文件夹里所有子级文件的大小总和，数据量很大，子级里还有子级(递归)怎么设计。
数据量大+深层递归，核心还是预计算方案，但要优化写入时的祖先更新策略。
物化路径+预计算total_size：
每个文件夹维护total_size（包含所有后代文件的大小总和，不只是直接子级）。文件增删时，从path里解析出所有祖先ID，批量更新。
```
文件结构：A/C/D/E/B.txt (100KB)
path: /A/C/D/E/B.txt
祖先：A、C、D、E 四个文件夹的total_size都要+100KB
```
数据量大时的问题和解决：
问题一，祖先层级深更新行数多：
一个文件嵌套20层，上传一个文件要更新20行。但这是常数级别的，跟文件总数无关，20次update对DB来说不算什么。用一条SQL批量更新：
```sql
UPDATE cloud_file 
SET total_size = total_size + #{fileSize}, version = version + 1
WHERE id IN (从path解析出的所有祖先ID列表);
```
问题二，高并发上传同一文件夹：
多人同时往同一个文件夹上传文件，祖先节点的total_size并发更新。不用乐观锁（会大量重试），直接用MySQL的原子操作total_size = total_size + ?，行锁持有时间极短，并发度够用。
问题三，文件夹移动（最复杂的场景）：
把文件夹D从A/C/下移动到A/F/下，要做三件事：
1. 更新D及所有后代的path（REPLACE批量改）
2. 旧祖先（A、C）的total_size减去D的total_size
3. 新祖先（A、F）的total_size加上D的total_size
```sql
-- 1. 批量更新path
UPDATE cloud_file SET path = REPLACE(path, '/A/C/D/', '/A/F/D/') 
WHERE path LIKE '/A/C/D/%';
-- 2. 旧路径祖先减
UPDATE cloud_file SET total_size = total_size - #{dTotalSize} WHERE id IN (C的ID);
-- 3. 新路径祖先加
UPDATE cloud_file SET total_size = total_size - #{dTotalSize} WHERE id IN (F的ID);
-- A是公共祖先不用动
```
移动操作放在一个事务里，保证原子性。文件夹下文件多时path更新行数大，可以异步+分批处理，移动期间文件夹标记为"迁移中"禁止操作。
问题四，数据量极大（亿级文件）：
- cloud_file表按用户ID分库，同一用户的文件在同一个分片，path查询不跨库
- total_size的更新走MQ异步，允许短暂不一致，定时任务对账修正
- 查询total_size直接读字段，O(1)，不受文件数量影响
兜底：定时任务全量重算total_size，跟预计算值比对，不一致的修正。防止长期运行后累积误差。

## 40. 实现一个全局发号器，供几百个应用，上千个服务使用，怎么实现？
几百个应用上千个服务，对发号器的要求是：高可用、高性能、全局唯一、趋势递增。
方案选型：号段模式+本地缓存，参考美团Leaf-Segment。
核心设计：
DB存号段分配表：
```sql
CREATE TABLE id_alloc (
    biz_tag VARCHAR(128) PRIMARY KEY,
    max_id BIGINT NOT NULL,
    step INT NOT NULL DEFAULT 1000,
    version INT NOT NULL,
    update_time TIMESTAMP
);
```
发号流程：
1. 服务启动时从DB批量取一段ID（比如step=1000），本地缓存[max_id+1, max_id+1000]这个区间
2. 后续发号直接从本地内存取，不访问DB，AtomicLong自增，极快
3. 本地号段快用完时（剩余20%），异步去DB取下一段，双buffer无缝切换
```sql
-- 取号段，乐观锁防并发
UPDATE id_alloc SET max_id = max_id + step, version = version + 1 
WHERE biz_tag = 'order' AND version = #{oldVersion};
```
双Buffer设计（关键）：
每个业务维护两个号段buffer，当前buffer用到20%剩余时，异步线程去DB加载下一个buffer。当前buffer用完直接切到下一个，零等待。避免了取号段时的DB延迟影响发号性能。
高可用：
- DB做主从+半同步，号段表数据量极小（每个业务一行），主从同步几乎无延迟。主挂了从升主，号段数据不丢。
- 发号服务多实例部署，每个实例各自取号段，互不冲突。实例A取[1,1000]，实例B取[1001,2000]。
- 某个实例挂了，它缓存的未用完的号段浪费掉（不连续但不重复），可接受。
- DB完全不可用时，本地buffer里还有号段可用，撑到DB恢复。
性能：
- 本地发号是内存操作，单机百万QPS没问题
- DB访问频率 = 总QPS / step，step设1000的话，100万QPS只需要1000次/秒DB访问
- 上千个服务同时取号段，DB压力 = 服务数 / (step / 单服务QPS)，很小
动态步长：根据业务QPS动态调整step，QPS高的业务step大（减少DB访问），QPS低的step小（减少ID浪费）。
跟雪花算法的对比：号段模式ID趋势递增对DB索引友好，不依赖时钟，没有时钟回拨问题。缺点是依赖DB，但双buffer+多实例基本消除了这个风险。

## 41. 实现一个全局发号器，供几百个应用，上千个服务使用，基于雪花算法，怎么保证高可用、性能？
雪花算法本地生成不依赖外部服务，天然高性能，核心要解决的是workerId分配和时钟回拨两个问题。
ID结构（64位）：
```
0 | 41位时间戳 | 10位workerId | 12位序列号
```
- 41位时间戳：支持69年
- 10位workerId：支持1024个节点，够上千个服务用
- 12位序列号：同一毫秒内支持4096个ID，单节点QPS上限409.6万
workerId分配（最关键）：
上千个服务实例，每个要有唯一的workerId，不能手动配置。
方案一，ZooKeeper分配：服务启动时在ZK创建临时顺序节点/snowflake/workers/0001，节点序号就是workerId。服务下线临时节点自动删除，workerId回收。美团Leaf-Snowflake就是这个方案。
方案二，DB分配：worker_id表记录每个实例的IP+端口和分配的workerId，启动时注册，心跳续期，超时回收。比ZK简单，不引入额外组件。
方案三，容器化场景用K8s的StatefulSet序号，Pod名自带序号（pod-0、pod-1），直接当workerId。
时钟回拨处理：
NTP同步可能导致时钟回拨，同一毫秒生成重复ID。
```java
if (currentTimestamp < lastTimestamp) {
    long offset = lastTimestamp - currentTimestamp;
    if (offset <= 5) {
        // 小幅回拨，等一等
        Thread.sleep(offset << 1);
    } else {
        // 大幅回拨，拒绝发号，告警
        throw new RuntimeException("时钟回拨超过5ms");
    }
}
```
更好的方案：启动时从ZK读取上次的时间戳，如果当前时间小于上次时间戳就拒绝启动。运行中定时上报时间戳到ZK，做持续校验。
高可用设计：
雪花算法本地生成，不依赖任何远程服务（ZK只在启动时用一次取workerId），所以天然高可用。ZK挂了不影响已启动的服务发号，只影响新服务启动注册workerId。
workerId缓存到本地文件，ZK不可用时从本地文件读取上次的workerId继续用。
性能优化：
- 发号是纯内存操作，AtomicLong自增，单机400万+QPS
- 不需要网络IO，不需要加锁（单线程或CAS）
- 批量发号：一次请求返回一批ID，减少调用次数
部署模式两种选择：
- 嵌入式SDK：每个服务引入jar包本地生成，零网络开销，性能最好。上千个服务各自生成，workerId保证不重复就行。
- 独立服务：发号器作为独立微服务，其他服务RPC调用。多了一次网络IO但统一管理方便。上千个服务调用的话要做好限流和连接池。
推荐嵌入式SDK模式，上千个服务各自本地生成，性能最高，没有单点风险。

## 42. 实现一个全局发号器，供几百个应用，上千个服务使用，基于MYSQL，怎么保证高可用、性能？
基于MySQL做发号器，核心是号段模式，不能每次发号都查DB，那性能扛不住。
基础方案：号段批量取号
```sql
CREATE TABLE id_generator (
    biz_tag VARCHAR(128) PRIMARY KEY,
    max_id BIGINT NOT NULL,
    step INT NOT NULL DEFAULT 1000,
    version INT NOT NULL,
    update_time TIMESTAMP
);
```
每次从DB取一个号段[max_id+1, max_id+step]缓存到本地内存，本地AtomicLong自增发号，用完再取下一段。DB访问频率从每次发号降到每step次一次。
高性能设计：
双Buffer预加载：维护两个号段buffer，当前buffer用到20%时异步加载下一个buffer。切换零等待，发号不阻塞。
```java
// 当前buffer剩余不足20%，触发异步加载
if (currentBuffer.remaining() < step * 0.2 && !nextBuffer.isReady()) {
    asyncLoadNextSegment();
}
// 当前buffer用完，无缝切到下一个
if (currentBuffer.isEmpty()) {
    swap(currentBuffer, nextBuffer);
}
```
动态步长：根据上次号段消耗速度动态调整step。消耗快的业务step调大（减少DB访问），消耗慢的step调小（减少ID浪费）。比如上次1000个号15分钟用完，下次step调到2000。
高可用设计：
DB主从+半同步：号段表数据量极小（每个业务一行），主从同步几乎无延迟。主挂了从升主，号段数据不丢。
多DB实例交替发号：部署两个DB实例，一个发奇数号段一个发偶数号段。
```sql
-- DB1: step=1000, 起始偏移0
-- 发号段: [1,1000], [2001,3000], [4001,5000]...
-- DB2: step=1000, 起始偏移1000  
-- 发号段: [1001,2000], [3001,4000], [5001,6000]...
```
一个DB挂了另一个继续发，ID不连续但不重复。Tinyid（滴滴开源）就是这个方案。
发号服务多实例：发号服务无状态部署多个实例，每个实例各自从DB取号段，互不冲突。某个实例挂了未用完的号段浪费掉，不影响其他实例。
本地缓存兜底：号段缓存到本地文件，DB完全不可用时从本地文件读取上次的号段继续用（有限的），争取时间等DB恢复。
性能数据：
- 本地发号：纯内存操作，百万QPS
- DB访问频率：假设总QPS 100万，step=1000，DB只需要1000次/秒
- 上千个服务同时取号段：每个服务QPS不同，DB总访问量 = 各服务QPS之和 / step，通常几百到几千次/秒，MySQL轻松扛住
跟Redis方案的对比：MySQL号段模式不依赖Redis，少一个组件。Redis INCR每次发号都要网络IO，MySQL号段模式本地发号零IO。MySQL数据持久化更可靠，Redis有丢数据风险。

## 43. 在不同的机房下，Snowflake算法产生的系统时钟回拨怎么处理
跨机房时钟回拨比单机房更复杂，因为不同机房的NTP源可能不同，时钟偏差更大。
分级处理策略：
小幅回拨（≤5ms）：直接等待，sleep到lastTimestamp追上来。跨机房NTP同步通常是毫秒级微调，等一下就好。
```java
if (offset <= 5) {
    Thread.sleep(offset << 1);
    currentTimestamp = System.currentTimeMillis();
    if (currentTimestamp < lastTimestamp) {
        throw new RuntimeException("等待后仍然回拨");
    }
}
```
中幅回拨（5ms-100ms）：借用序列号的扩展位。把workerId的10位拆成5位机房ID+3位机器ID+2位回拨位。每次检测到回拨，回拨位+1（0→1→2→3），相当于切换到一个"新的虚拟节点"，生成的ID跟回拨前不冲突。最多支持3次回拨。
```
原来: 时间戳 | 机房(5) | 机器(3) | 回拨位(2) | 序列号(12)
回拨后: 同一时间戳但回拨位不同，ID不重复
```
大幅回拨（>100ms）：拒绝发号，告警人工介入。大幅回拨说明NTP配置有问题或者机器时钟硬件故障，不能靠软件兜底。
跨机房特有的处理：
每个机房独立的workerId段：机房A用workerId 0-511，机房B用512-1023。不同机房的时钟即使不同步，生成的ID也不会冲突，因为workerId不同。时钟回拨只影响本机房内的ID唯一性。
上报时间戳到ZK/DB：每个节点定时上报自己的时间戳到ZK。启动时读取上次上报的时间戳，如果当前时间小于上次时间戳就拒绝启动。运行中检测到回拨也上报告警。
独立NTP源：每个机房部署自己的NTP Server，所有机器同步到本机房NTP，减少跨机房NTP同步的偏差。NTP Server之间做对时校准。
兜底方案：如果时钟问题频繁，该机房的发号降级为号段模式从DB取号，不依赖时钟。等时钟稳定后再切回雪花算法。

## 44. 假如此时扩容一台新的机器，对于产生ID的影响？
扩容新机器对ID生成的影响取决于用的是哪种方案。
雪花算法：
核心影响是workerId分配。新机器启动需要拿到一个唯一的workerId，不能跟已有机器重复。
如果用ZK分配：新机器启动时在ZK创建临时顺序节点，自动拿到新的workerId，跟已有节点不冲突。对已有机器零影响，无需重启。
如果用DB分配：新机器注册一条记录拿到新workerId，同样不影响已有机器。
如果用配置文件手动配：扩容时要人工分配workerId，容易出错重复，不推荐。
潜在风险：10位workerId最多支持1024个节点。如果已经用了1000个，扩容空间只剩24个。要提前规划好workerId的位数分配，或者用回收机制把下线机器的workerId释放给新机器。
号段模式：
完全无影响。新机器启动后从DB取自己的号段，跟已有机器各取各的，互不干扰。号段模式不关心有多少台机器，每台各自取号段就行。
唯一要注意的是DB的并发取号段压力。原来1000台机器，扩容到1100台，DB取号段的频率增加10%，通常可以忽略。
MySQL自增ID：
如果用的是MySQL auto_increment做发号，扩容新机器没有直接影响，因为发号都走同一个DB。但如果是多DB实例交替发号（奇偶方案），扩容新DB实例需要重新规划步长和偏移量，比较麻烦。
总结：雪花算法扩容要关注workerId分配不重复，号段模式扩容基本无感知。生产上推荐用ZK或DB自动分配workerId，扩容时自动注册，不需要人工干预。
