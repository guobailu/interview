# 抖音电商 Q&A

## 1. 商品运营侧的研发如何更好的支持商品营销侧
核心是商品运营侧把数据和能力标准化输出，让营销侧低成本接入。
数据层面：建商品数据中心，把SPU/SKU/类目/属性/品牌/标/CPV聚合成统一商品视图，营销侧一次调用拿到完整画像，不用直接查DB。商品变更通过MQ实时推送，上下架、改价、库存变化营销侧实时感知。
能力层面：提供可编排的标准接口。批量打标能力，营销做活动直接调接口给商品打"618特惠"标，不用排期开发。圈品引擎，按类目+属性+价格+品牌多维度圈选商品，底层走ES检索。价格管控，营销改价时商品侧做校验兜底，防止改出负数或低于成本价。
协同机制：商品变更走事件总线发领域事件，营销侧按需订阅，两边解耦。商品侧不用关心下游有多少消费方。
之前在天猫好房做商品中台重构就是这个思路，标准化后营销团队的需求交付从两周降到两天，大部分活动配置自助完成。

## 2. 负责商品的研发团队和营销团队协作时，可能出现的问题
几个常见的坑：
数据口径不一致：商品侧的"在售商品"和营销侧理解的"可参与活动的商品"定义不同。比如商品状态是在售但库存为0，商品侧算在售，营销侧觉得不能参加活动。这种口径差异不提前对齐，活动上线就出问题。解决方案是建立统一的商品状态枚举和业务语义文档，双方共同维护。
变更通知不及时：营销活动已经配好了，商品侧突然下架了某个SKU或者改了类目，营销侧不知道，活动页面展示了已下架的商品。必须靠事件驱动，商品变更实时推MQ通知营销侧，营销侧做好兜底校验，展示前再check一次商品状态。
排期冲突：大促前营销侧需要商品侧配合做批量打标、圈品、改价，但商品侧自己也有迭代排期。两边都觉得自己的需求优先级高。要提前建立大促联合排期机制，大促前一个月锁定需求，商品侧预留20%的buffer给营销侧的临时需求。
接口性能拖累：营销活动瞬时流量大，商品接口扛不住导致活动页加载慢。商品侧要提前做好缓存预热和限流降级，核心商品数据走本地缓存+Redis多级缓存，不能让营销的流量直接打穿到DB。
权责不清：商品价格出了问题，是商品侧的锅还是营销侧改价改错了？要有完整的操作审计日志，谁在什么时间改了什么，出了问题能追溯。

## 3. 电商类目属性系统怎么设计
类目属性是电商的地基，搜索、推荐、营销、交易都依赖它。
类目设计：前后台类目分离。后台类目稳定，按商品本质分（生鲜→水果→苹果），运营管理用。前台类目灵活，按用户场景组织（今日特惠→时令水果），可以随时调整映射关系，大促临时挂新导航不动后台数据。类目表用parent_id+path物化路径存储，path方便like查所有子类目，避免递归。
属性模型：三层——属性组、属性、属性值。属性分三类：关键属性决定SPU（品牌+品名），销售属性决定SKU（规格+包装），非关键属性做描述（产地、保质期）。到家业务特殊的是要加履约属性，比如配送温区（常温/冷藏/冷冻）、保质期时效，这些直接影响库存和配送。
CPV体系：Category-Property-Value，类目绑定属性集，不同类目挂不同属性。叶子类目继承父类目属性可覆盖。属性值统一管理防止同义词泛滥（"500ml"和"500毫升"要归一）。枚举值走标准化，输入值做校验。
存储：类目和属性定义走MySQL，商品属性值用EAV或JSON灵活存储。搜索场景同步到ES支持属性筛选聚合。类目树全量缓存Redis，变更MQ广播失效。
之前在天猫好房做的类目属性体系跟这个思路一样，房产的类目（新房/二手/租房）、属性（户型/面积/朝向）映射到到家业务就是品类和商品规格，底层模型通用。

## 4. 假设是商品中台的开发，高并发海量数据下，所有的商品数据放在一张表，造成性能瓶颈，要做一张表的垂直拆分，老表已经承担读写流量了，怎么平滑稳定的迁移
分四步走，全程可回滚。
第一步，新表建好+双写：先按垂直拆分规则建好新表（比如商品基础表、商品扩展表、商品媒体表），应用层改造写入逻辑，写老表的同时双写新表。双写用同一个本地事务保证原子性，新表写失败不影响老表，记日志后补偿。同时用Canal监听老表binlog，把存量数据同步到新表。
第二步，数据校验：全量对账工具比对老表和新表数据，逐字段校验，不一致的修复。增量对账持续跑，确保双写期间数据完全一致。这个阶段读还是走老表，新表只写不读。
第三步，灰度切读：通过配置开关按比例把读流量切到新表，先1%→10%→50%→100%。每个阶段观察RT、错误率、业务指标，有问题秒级切回老表。到家业务的商品读QPS很高，切流量时要关注缓存命中率和DB连接数。
第四步，停写老表：读全部切到新表且稳定运行一周后，停止双写老表，写也全部走新表。老表保留一个月做兜底，确认没问题后下线。
关键细节：双写期间如果新表结构跟老表差异大（字段拆分），要在应用层做数据转换适配，用DAO层的策略模式切换读写目标，业务代码不感知。上下游团队（搜索、详情、营销）如果直接消费商品数据变更消息，消息格式要保持兼容，先发老格式+新格式双发，下游逐步切换。

## 5. 假设你是IC商品中台的开发，设计一套详细的表模型支持当前并承载未来海量激增的商品，并且要支持未来的扩展，你怎么设计这个表模型？(需要考虑到类目、CPV、SKU、SPU、品牌、库存、标、扩展属性、视频、图片、图文详情等，结合主子商品表、主子表关联关系表、扩展表、标、库存表)店中店商品、主子商品、商品关系等，主子表、主子表关联关系表、扩展表、标、库存表，通过打标、扩展属性实现底层数据模型的扩展性。
整体按DDD思路拆成几个核心域的表模型：
商品主表（spu_info）：商品ID、商品名称、品牌ID、后台类目ID、前台类目ID、商品类型（普通/主子/店中店）、状态、版本号。这是SPU级别，一个商品一条记录。
SKU表（sku_info）：SKU_ID、SPU_ID、销售属性组合（JSON）、价格、市场价、状态、版本号。SPU和SKU是1:N关系。
主子商品关联表（spu_relation）：关系ID、主商品ID、子商品ID、关系类型（主子/套装/搭配/店中店）、排序号、版本号。店中店就是一种关系类型，主商品是店铺维度的虚拟SPU，子商品是实际SKU。
类目属性表（category_property）：类目ID、属性ID、属性类型（关键/销售/非关键）、是否必填、排序号。CPV体系的核心，类目绑定属性集。
商品属性值表（spu_property_value）：SPU_ID、属性ID、属性值ID/属性值文本、版本号。EAV模型，灵活存储不同类目的差异化属性。
扩展属性表（spu_extension）：SPU_ID、扩展key、扩展value（JSON）、版本号。兜底所有未来新增的非标属性，不用改表结构。比如到家业务的配送温区、保质期这些后加的字段都走扩展表。
标表（spu_tag）：SPU_ID/SKU_ID、标类型（系统标/运营标/营销标）、标值、生效时间、失效时间、版本号。打标实现业务扩展，比如"新品""爆款""618特惠"都是标，营销侧通过标圈品。
库存表（sku_stock）：SKU_ID、仓库ID、可用库存、冻结库存、版本号。按SKU+仓维度管理，到家业务还要加门店ID维度。库存独立表，不跟商品表混，扣减用乐观锁。
媒体表（spu_media）：SPU_ID/SKU_ID、媒体类型（主图/视频/图文详情）、URL、排序号、版本号。图片视频独立存储，URL指向OSS。
品牌表（brand_info）：品牌ID、品牌名、logo、状态。独立维护，SPU通过品牌ID关联。
扩展性保障：新业务需求优先走扩展属性表和标表，不改主表结构。海量数据按商品ID分库分表，主子关系表按主商品ID分片保证同一组主子商品在同一个分片。

## 6. 理论上都应该IC做，但中台会随着业务发展，效率变差(我理解商品中台应该是要求更高一些，所以要多招人)，天猫好房的商品后台也是基于这个背景做整个房产商品的全生命周期的操作，包含《房产商品后台录入房产信息整体的业务流程》，及商品的审核、发布更新、上下架。。。过程中也会涉及商品地理位置、价格等个性化定制。
对，中台随着业务膨胀效率下降是必然的，核心原因是中台要兼顾通用性，但每个业务线的个性化需求越来越多，排期就卡住了。
天猫好房的商品后台就是在这个背景下做的。房产商品的全生命周期流程：开发商/中介在后台录入楼盘信息（基础信息→地理位置→户型图→价格→配套设施→VR数据），走审核流（系统自动校验+人工审核），审核通过后发布上架，运营可以随时更新价格、调整上下架状态、修改图文详情。
个性化定制是关键挑战。房产商品跟标品不一样，地理位置是核心属性要做地图选点和行政区划关联，价格不是固定的而是区间价（总价/单价/首付），还有学区、交通、周边配套这些非标属性。这些在中台的标准商品模型里是没有的，硬塞进去会污染中台模型。
我们的做法是中台提供标准能力+扩展点机制。标准的商品发布、审核、上下架流程走中台，房产特有的地理位置、价格模型、个性化属性走扩展表和扩展接口。中台暴露SPI扩展点，房产业务方实现自己的校验逻辑和数据处理逻辑，插到中台的流程里。这样中台不用为每个业务线改代码，业务方也能灵活定制。
映射到抖音电商到家业务也是一样的思路，到家商品有配送范围、时效、温区这些个性化属性，不应该改中台主模型，走扩展属性+扩展点就行。

## 7. 抖音电商和美团电商的商品数据模型各有什么优劣势（基于SKU、SPU、库存、类目、商品属性、标、品牌，CPV体系的对比）
两家的商品模型底层都是SPU/SKU体系，但因为业务形态不同，侧重点差异很大。
抖音电商：内容驱动型，商品模型要强关联内容。SPU/SKU标准化程度高，CPV体系完善，类目属性管控严格，因为要支撑搜索推荐和直播间的商品卡展示。优势是标准化做得好，同一个SPU下不同商家的SKU能聚合比价，有利于用户决策。品牌和标体系丰富，支持各种营销标（直播专享、限时折扣）。劣势是到家业务的本地化属性（配送范围、门店库存、时效）在原有模型里支持不够，需要扩展。库存模型偏中心化，到家业务需要门店级库存，跟传统电商的中心仓库存模型有冲突。
美团电商：LBS驱动型，商品模型天然围绕门店和地理位置设计。SKU跟门店强绑定，同一个商品在不同门店可以有不同价格和库存，这是到家业务的核心需求。优势是门店级库存管理成熟，配送范围、时效、起送价这些本地化属性是一等公民，不是扩展字段。类目体系按本地生活场景设计（餐饮/生鲜/超市），贴合到家业务。劣势是SPU标准化程度不如抖音，同一个商品在不同门店可能是不同的SPU，商品聚合和比价能力弱。CPV体系相对简单，非标品多，属性管控松，导致商品质量参差不齐。
核心差异：抖音是"货找人"，商品模型服务于内容推荐，标准化优先；美团是"人找货"，商品模型服务于本地履约，本地化优先。抖音做到家业务的挑战就是要在标准化模型上叠加本地化能力，既要保持CPV体系的规范性，又要支持门店级的差异化。

## 8. 商品类目变更，类目从其中一个拆解成二个类目，怎么解决商品无损和一致性
类目拆分是高风险操作，影响面广（商品、搜索、营销、交易都依赖类目），必须分阶段平滑迁移。
第一步，新类目准备：先在类目树上创建两个新类目，配置好各自的属性集（CPV）。新类目先不生效，状态为"待启用"。制定商品分流规则，明确老类目下的商品按什么条件分到新类目A和新类目B，比如按品牌或属性值区分。
第二步，商品预分配：跑离线任务，按分流规则给老类目下每个商品打上"目标类目"标签，但不真正改类目字段。人工抽检确认分配结果，有争议的商品单独处理。这一步不影响线上任何功能。
第三步，双写过渡：启用新类目，商品同时挂在老类目和新类目上（双挂）。搜索、详情、营销等下游系统先按老类目消费，同时开始适配新类目。新发布的商品直接走新类目，存量商品双挂。
第四步，灰度切流：下游系统逐步从老类目切到新类目消费。搜索先切（影响最大但可回滚），然后详情、营销、交易依次切。每切一个下游观察一天，确认无异常再继续。
第五步，清理老类目：所有下游都切到新类目后，去掉商品的老类目挂载，老类目标记为"已废弃"。保留一个月兜底，确认没有遗漏的消费方后彻底下线。
一致性保障：整个过程商品数据不做删除只做新增和更新，任何一步出问题都能回退。类目变更事件通过MQ通知所有下游，下游做好兼容，老类目ID和新类目ID都能识别。

## 9. 很多下游系统都是依赖类目做的配置，回滚的时候下游系统怎么才能没有影响
核心是回滚时老类目一直没下线，下游的老配置一直在生效。
关键设计：双挂期间老类目不动、不删、不改。新类目是"加"上去的，不是"替换"老类目。所以回滚就是把新类目的挂载去掉，商品回到只挂老类目的状态，下游系统还是按老类目的配置跑，完全无感。
下游系统的配置兼容策略：切流不是让下游删掉老类目配置换成新类目配置，而是新老配置并存。下游按优先级读：先查新类目配置，有就用新的；没有就fallback到老类目配置。这样回滚时只需要关掉新类目的配置开关，自动fallback到老配置，不用改任何东西。
具体实现：搜索侧，新老类目的索引映射同时存在，通过配置开关决定用哪个。营销侧，活动配置同时绑定老类目ID和新类目ID，回滚时新类目ID的配置不生效就行。交易侧，类目校验逻辑做兼容，老类目ID和新类目ID都认。
所以整个过程下游不需要做"回滚操作"，只需要商品侧把新类目挂载去掉+关掉新类目的配置开关，下游自动回到老状态。这就是为什么双挂过渡期很重要，它保证了任何时刻都能无损回退。

## 10. 设计引入AI对商品发布更新等操作做自动化的审核和诊断，提高整个商品发布上新的时效。
核心是AI做前置自动审核+智能诊断，把人工审核从"全量审"变成"抽检复核"，大幅缩短上新时效。
审核链路：商品提交 → 规则引擎(硬规则) → AI审核(软规则) → 自动通过/人工复核/自动驳回。
第一层规则引擎秒级过滤：必填字段缺失、价格异常（为0或负数）、违禁词命中、类目不匹配，这些确定性规则不需要AI，Drools规则引擎毫秒级返回，过滤掉30%的明显不合格商品。
第二层AI审核并行处理：图片审核用视觉模型检测主图质量（模糊、水印、牛皮癣）、是否盗图、违规内容。文本审核用大模型分析标题描述，检测虚假宣传和夸大用语。属性诊断用RAG检索同类目商品属性分布，偏离太大的标记异常。价格诊断跟同类目历史价格对比，异常偏高偏低的标可疑。图片和文本并行推理，总耗时控制在10秒内。
审核结果三档：置信度>95%且无风险自动通过，中等风险自动通过但标记待抽检，明显风险自动驳回并附AI给出的修改建议。
智能诊断提升上新效率：AI不只说"不通过"，还告诉商家怎么改。比如"标题含夸大用语建议修改"、"主图清晰度不足建议上传800×800以上"。商家按建议改完再提交，一次通过率从60%提升到90%。
技术实现：AI推理走异步，商品提交后发MQ触发，审核完回调更新状态，不阻塞发布流程。人工复核结果反馈给模型做fine-tune持续优化。AI服务挂了降级为纯规则引擎+全量人工复核。

## 11. 放弃支持电商房产业务的商品研发，选择抖音到家业务系统的研发的理由？
说实话不是放弃，是选择更有成长空间的方向。
业务层面：房产电商受政策和市场周期影响太大，2023年以来整个房地产行业下行，天猫好房的业务规模在收缩，技术团队能做的事情越来越有限。而抖音到家业务正处于高速增长期，即时零售是万亿级赛道，用户需求强、增长空间大，技术挑战也更多。从职业发展角度，选一个上升期的业务比守一个收缩期的业务更有价值。
技术层面：房产商品是低频高客单价，系统的并发量和数据量有天花板。到家业务是高频刚需，商品SKU量级、订单量级、流量峰值都比房产大几个数量级，对系统的高并发、高可用、实时性要求更高。我在天猫好房积累的商品中台、类目属性、CPV体系的经验可以直接迁移到到家业务，但到家业务还有门店级库存、配送时效、直播带货这些新挑战，能让我的技术能力再上一个台阶。
个人层面：我希望做的事情能被更多用户感知到。房产交易一年可能就几万笔，到家业务每天几百万单，做的每一个优化都能直接影响几百万用户的体验，这种成就感完全不一样。

## 12. 即时零售自营模式
即时零售自营模式就是平台自己采购商品、自己管库存、自己配送，全链路自控。典型代表是朴朴超市、叮咚买菜、美团买菜。
核心链路：平台自建前置仓（离用户3-5公里），自己向供应商采购商品入仓，用户下单后从最近的前置仓拣货，自有骑手30分钟送达。
商品模型特点：SKU由平台统一管理，不是商家上传的。商品的类目、属性、价格、图片都是平台运营维护，标准化程度高。库存是仓级库存，同一个SKU在不同前置仓库存独立，要做到仓维度的实时库存管理。
优势：品控好，商品质量平台自己把关；配送快，自有骑手+前置仓保证时效；用户体验一致，不存在不同商家服务参差不齐的问题。
劣势：重资产，要建仓、养骑手、压货，资金压力大；SKU有限，受仓面积限制一般3000-5000个SKU，远不如平台模式丰富；损耗高，生鲜品类过期损耗是大成本。
跟平台模式（抖音到家、美团外卖）的区别：平台模式是商家管商品管库存，平台做撮合和配送。自营模式是平台全包。抖音到家目前主要走平台模式，但理解自营模式的供应链和库存逻辑，对设计到家业务的商品和库存系统很有帮助。
