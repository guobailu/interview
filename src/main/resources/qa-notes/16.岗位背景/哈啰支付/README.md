# 哈啰支付 Q&A

## 1. 四轮出行行程、计价、支付怎么设计
四轮出行的行程、计价、支付三个域要解耦设计，通过事件驱动串联。
行程域：
用户叫车→匹配司机→司机接单→行程开始→行程中（实时上报GPS轨迹）→行程结束。状态机严格流转：待接单→已接单→已到达→行程中→已完成→已取消。每个状态变更发领域事件通知下游。轨迹数据高频上报（每秒一次），写入Kafka异步存HBase，Redis存最新位置供实时查询。
计价域：
行程结束触发计价。计价引擎用策略模式，支持多种计价规则：起步价+里程费+时长费+远途费+夜间加价+动态调价。计价因子从行程域拿（总里程、总时长、起终点），从配置中心拿（城市费率、车型系数）。计价结果要可追溯，每笔订单存完整的计价明细（每段里程单价、每段时长单价），方便用户投诉时核查。动态调价用Flink实时计算供需比，供不应求时上浮系数。
支付域：
计价完成触发支付。资金安全三要素必须做到：
1. 分布式锁：按订单号加锁，防止同一笔行程重复发起支付。用Redis setnx，锁粒度到订单级别。
2. 幂等：支付请求带唯一支付流水号，支付网关用幂等表去重，重复请求直接返回上次结果。
3. 状态机：待支付→支付中→已支付→已退款，严格校验前置状态，已支付的不能再发起支付。
支付走聚合支付网关，对接支付宝/微信/哈啰钱包。先冻结用户余额或预授权（TCC的Try），行程结束确认金额后Confirm扣款，取消行程Cancel解冻。支付结果异步通知行程域更新订单状态，通知失败走定时查询兜底。T+1对账比对支付流水和行程订单，不一致的走差错处理。

## 2. 设计一个秒杀系统
秒杀核心是层层过滤，把请求量从百万级削减到DB能承受的量级。
前端层：页面静态化推CDN，倒计时客户端本地计算不轮询。按钮点一次灰掉防重复提交，加滑块验证码拦机器人。
网关层：Nginx按IP限流每秒1次，网关按用户ID限流防刷。未登录、黑名单用户直接拦掉。这一层过滤掉80%的无效请求。
服务层：Redis预扣库存是核心。库存预加载到Redis，用DECR原子扣减，返回值<0说明没库存直接拒绝。一条命令搞定，不需要分布式锁，单key十万QPS。扣减成功的请求发MQ异步创建订单，用户端返回"抢购成功，请在15分钟内支付"。
订单层：消费MQ创建订单，资金安全三要素：
1. 分布式锁：按用户ID+活动ID加锁，同一用户同一活动只能下一单。
2. 幂等：用唯一请求ID做幂等键，MQ重复消费不会重复创建订单。
3. 状态机：待支付→支付中→已支付→已完成/已超时关闭，严格校验前置状态。
支付超时15分钟自动关单，库存回到Redis（INCR加回去），触发候补通知。
防超卖双保险：Redis DECR是第一道，DB层update stock set count=count-1 where count>=1是第二道。Redis和DB最终一致靠MQ异步同步+定时对账。
高可用：Redis主从+Sentinel，秒杀服务多实例无状态。Redis挂了直接返回"系统繁忙"，不降级到DB防超卖。

## 3. 应该在什么时间扣除库存
三种时机各有利弊，看业务场景选。
下单减库存：用户点下单就扣库存。优势是不会超卖，逻辑简单。劣势是用户下单不付款占着库存，恶意下单能把库存锁死。秒杀场景用这个，因为秒杀商品稀缺，抢到就是抢到，配合超时自动释放（15分钟不付款关单加回库存）。
支付减库存：用户付款成功才扣库存。优势是不存在占库存不付款的问题。劣势是可能超卖，多人同时下单都显示有库存，但付款时库存不够了，用户体验差。适合库存充足的普通商品。
预扣减库存（推荐）：下单时先冻结库存（可用库存减、冻结库存加），支付成功后确认扣减（冻结库存减），超时未支付释放（冻结库存减、可用库存加）。本质就是TCC模式，Try冻结、Confirm确认、Cancel释放。
哈啰的出行场景比较特殊，不存在传统意义的库存，但类似的概念是"运力"。叫车时预占一个司机（冻结运力），行程完成确认消耗，取消订单释放运力。思路跟预扣库存一样。
秒杀场景用下单减库存+超时释放，普通电商用预扣减库存，库存充足的标品可以用支付减库存。核心是根据超卖容忍度和用户体验来选。

## 4. 对于热点商品的秒杀，打到同一个消息分区。消息分区压力比较大，怎么解决
热点商品的消息都按商品ID路由到同一个分区，单分区消费能力有上限，堆积就慢了。
方案一，打散路由key：不用商品ID做路由key，改用订单ID或用户ID。同一个商品的消息分散到多个分区并行消费。代价是同一商品的消息顺序性丢了，但秒杀场景下单操作本身是幂等的，顺序性不是强需求。
方案二，分区内多线程消费：路由key不变还是商品ID，但消费者内部用线程池并行处理。从分区拉一批消息，按订单ID分组丢到线程池并行执行，同一订单的消息还是串行保证顺序。相当于在消费者内部做了二次分发。
方案三，热点商品单独Topic：提前识别热点商品（大促前已知哪些是秒杀品），给热点商品单独建Topic，分区数设多一些（比如64个），消费者实例也多部署。普通商品走默认Topic。路由层根据商品ID判断走哪个Topic。
方案四，前置过滤减少消息量：秒杀场景下Redis DECR已经过滤掉了大部分请求，只有扣减成功的才发MQ。如果库存100件，最多就100条消息进MQ，单分区完全扛得住。所以核心是在MQ之前把量削下来，而不是让MQ扛全量。
实际推荐方案四+方案一组合：Redis预扣大幅削量，剩余的消息用订单ID路由打散到多分区，基本不会有分区压力问题。

## 5. 对于热点商品的秒杀，数据库压力比较大，怎么解决
核心思路是让请求尽量不打到DB，打到DB的也要分散热点。
第一层，Redis挡住读和预扣：库存查询和扣减全走Redis DECR，不查DB。库存100件，最多100个请求能过Redis这关，后面的全部拒绝。DB的压力从百万QPS降到100次。
第二层，MQ削峰控速：过了Redis的请求进MQ，消费者按DB能承受的速率消费（比如每秒50个），不是一股脑全打到DB。DB的瞬时并发被MQ拉平了。
第三层，库存拆桶分散行锁：同一个SKU的库存拆成N个子记录（比如100件拆成10个桶每个10件），写请求hash到不同桶，把单行热点锁分散成10行，并发度提升10倍。某个桶扣完了再去其他桶借调。
第四层，DB层面优化：库存扣减用乐观锁update stock set count=count-1 where sku_id=? and count>=1，不用select for update悲观锁。事务尽量短，只包含扣减操作，订单创建和流水记录异步写。库存表独立库，不跟商品表混，避免大事务锁表。
第五层，异步落库：订单创建走MQ异步，批量insert减少DB交互。库存变更流水也异步写，不在扣减事务里。
兜底：Redis和DB双保险防超卖，定时对账比对Redis库存和DB订单数，不一致告警修复。

## 6. 异步减库存怎么保证一致性
异步减库存的一致性问题就是Redis扣了但DB没扣成功，或者MQ消息丢了。
消息可靠投递：Redis DECR成功后，用本地消息表保证消息一定发出去。在同一个线程里先写本地消息表（状态为待发送），再发MQ。发送成功更新状态为已发送，发送失败由定时任务扫描重发。这样即使MQ挂了消息也不会丢。
消费端幂等：消费者用订单号做幂等键，DB扣减前先查幂等表，已处理过的直接跳过。防止MQ重复投递导致多扣库存。
DB扣减兜底：消费者执行update stock set count=count-1 where sku_id=? and count>=1，影响行数为0说明库存已经被扣完了（极端情况Redis和DB不一致），这时候要把Redis的库存也修正为0，同时把这笔订单标记为失败退款。
消费失败处理：DB扣减失败自动重试，重试3次还失败进死信队列。死信队列的消息人工处理，同时把Redis的库存INCR加回去，释放给其他用户。
定时对账：每分钟跑一次轻量对账，比对Redis库存和DB已扣减的订单数。Redis库存 + DB已扣减数 = 初始库存，不等就告警。T+1全量对账做最终兜底。
超时释放：用户15分钟不付款，关单时Redis INCR加回库存，DB也回滚扣减。关单操作也要做幂等，防止重复释放。
核心原则：Redis做快速判断，DB做最终裁决，对账做兜底修正。允许短暂不一致，保证最终一致。

## 7. 分库分表情况下，热点商品在第一个库没有库存，但是在第二个库有库存怎么优化处理
这就是库存拆桶后的桶间调度问题，某个桶扣完了但其他桶还有。
方案一，Redis统一管控+DB分桶：Redis维护总库存做预扣判断，DB层面按桶分散写。Redis DECR总库存成功后，路由到具体桶扣减。如果路由到的桶没库存了，换下一个桶扣。Redis层面不关心哪个桶有库存，只管总量够不够。
```
Redis: total_stock = 100 (总库存)
DB桶1: count = 0 (扣完了)
DB桶2: count = 50 (还有)
```
扣减逻辑：Redis DECR total_stock成功 → 先尝试桶1 update where count>=1 → 影响行数0 → 尝试桶2 → 成功。
方案二，桶间借调：桶1库存为0时，从桶2转移一批库存过来（比如转20件）。用一个事务：桶2减20，桶1加20。这样后续请求路由到桶1也能扣成功，不用每次都跨桶查。借调操作可以异步做，定时任务检测各桶库存不均衡时触发。
方案三，动态路由：不固定hash到某个桶，而是维护一个"有库存的桶列表"在Redis里。扣减时从列表里随机选一个桶，桶扣完了从列表移除。这样请求天然只打到有库存的桶上，不会打空桶。
实际推荐方案一+方案三组合：Redis管总量，动态路由到有库存的桶，桶扣完自动摘除。简单高效，不需要复杂的桶间借调逻辑。

## 8. 会出现并发导致的超卖问题么
不会，因为每一层都有原子性保障。
Redis层：DECR是原子操作，Redis单线程执行，不存在两个请求同时读到相同值再各自减1的问题。100个库存，第101个DECR返回-1，直接拒绝。这一层不会超卖。
DB层：update stock set count=count-1 where sku_id=? and count>=1，这条SQL在InnoDB里是行锁+原子操作，count>=1的条件保证不会扣成负数。即使多个请求并发打到同一行，行锁串行化执行，最后一件库存只有一个请求能扣成功，其他的影响行数为0返回失败。这一层也不会超卖。
桶间调度会不会超卖：不会。Redis管总量，总量DECR到0就不再放请求进来。DB各桶独立扣减，每个桶自己的count>=1保证不超扣。桶1没了去桶2扣，桶2的count也有>=1保护。总量=各桶之和，不会出现总量超卖。
唯一可能出问题的场景：Redis和DB不一致。比如Redis扣了但DB扣减失败（消费者挂了），Redis库存比DB实际库存少，这时候不是超卖而是少卖（有库存但Redis认为没了）。通过定时对账修正Redis库存就行。
反过来Redis库存比DB多的情况不会出现，因为是先Redis DECR再DB扣减，不是反过来。所以最坏情况是少卖不是超卖，资金安全有保障。

## 9. 分布式锁会影响性能，如果不用分布式锁，可以用什么实现
几种替代方案，看场景选：
Redis原子操作：库存扣减用DECR，计数器用INCR，本身就是原子的不需要锁。Lua脚本可以把多步操作合成一个原子操作，比如"判断库存够不够+扣减"一个Lua搞定。单key十万QPS，比分布式锁快两个数量级。
DB乐观锁：update set count=count-1, version=version+1 where id=? and version=? and count>=1。不加锁，靠版本号CAS控制并发，影响行数为0说明被别人改了，重试就行。适合冲突不太频繁的场景，冲突率高了重试多性能也差。
DB唯一索引做幂等：用唯一索引代替分布式锁做防重。比如防止重复下单，订单表建(user_id, activity_id)唯一索引，重复插入直接报唯一冲突，不需要先加锁再判断。
MQ串行化：把并发请求按业务key路由到同一个分区，消费者单线程顺序处理，天然串行不需要锁。适合对实时性要求不高的场景，比如异步创建订单。
本地内存队列：同一个JVM内用BlockingQueue把并发请求排队，单线程消费。适合单机场景或者请求已经按key路由到固定机器的场景。
选型建议：库存扣减用Redis原子操作，防重复用DB唯一索引，异步处理用MQ串行化，简单的读写冲突用DB乐观锁。分布式锁只在确实需要跨资源协调的场景才用。

## 10. 网约车支付功能设计
网约车支付功能设计，分三个核心模块：
支付方式管理：支持多种支付渠道（支付宝/微信/哈啰钱包/银行卡），聚合支付网关统一封装，上层业务不感知具体渠道。用户可以设置默认支付方式，支持组合支付（钱包余额+微信补差价）。
支付流程：行程结束→计价引擎算出金额→发起支付。资金安全三要素：
1. 分布式锁：按行程单号加锁，防止同一笔行程重复发起支付。
2. 幂等：支付请求带唯一流水号，支付网关幂等表去重。
3. 状态机：待支付→支付中→已支付→已退款，严格校验前置状态。
先走预授权模式：叫车时冻结预估金额（TCC的Try），行程结束按实际金额扣款（Confirm），取消行程解冻（Cancel）。预授权金额不够的补扣差额，多冻结的退回。
异常处理：支付超时走定时查询确认最终状态，不能直接当失败退款。支付失败自动重试一次，还失败挂起等人工处理。司机端和乘客端的支付状态通过MQ异步同步，保证双方看到一致的结果。
清结算：T+1对账比对支付流水和行程订单。司机收入走结算系统，扣除平台佣金后按周/月打款到司机账户。结算走批量处理，Flink实时计算司机收入明细，定时任务触发批量打款。
