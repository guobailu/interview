# HRBP(带案例) Q&A

## 1. 怎么管理项目开发过程中的风险？
我一般从三个阶段来管控风险。
项目启动时做风险预判。比如在天猫好房商品中台重构的时候，我提前识别了几个高风险点：老系统数据迁移可能丢数据、新老系统并行期间的一致性问题、上下游十几个团队的接口改造排期不一致。针对每个风险我都提前准备了兜底方案，数据迁移做了全量对账工具，并行期间用双写+灰度切流，排期不一致的跟各团队leader提前锁定里程碑。
开发过程中做风险暴露。每天站会不只同步进度，重点同步风险和卡点。我要求团队成员遇到问题当天暴露，不要藏着等到最后一天才说做不完。有一次一个同学在做批处理框架的时候发现性能达不到预期，当天就暴露了，我们及时调整了技术方案，没有影响整体排期。
上线阶段做风险兜底。灰度发布、监控告警、回滚预案这些是标配。商品中台切流的时候我们是按1%→10%→50%→100%逐步放量的，每个阶段观察半天确认没问题再放大，最终零故障上线。

## 2. 研发怎么和非研发沟通，都有什么具体方法可以达到和其他部门有效沟通？
核心就是说对方听得懂的话，不要甩技术术语。
跟产品沟通：用业务语言。比如产品问"这个需求能不能做"，我不会说"要改数据模型加索引做缓存"，而是说"能做，大概需要两周，上线后搜索速度能从3秒降到0.5秒，用户体验会好很多"。把技术方案翻译成业务价值和时间成本，产品最关心这两个。
跟运营沟通：用数据和效果说话。在天猫好房的时候，运营经常提需求说"我要一个批量改价的功能"，我会先问清楚场景，然后告诉他"这个功能上线后，你原来改一千个房源要两小时，以后十分钟搞定"，运营一听就知道值不值得排期。
跟老板汇报：讲结果和影响。不要讲技术细节，讲"我们做了什么，带来了什么业务结果"。比如"商品中台重构完成，系统稳定性从99.5%提升到99.99%，大促期间零故障，支撑了XX亿GMV"。
具体方法：多用类比，把技术概念类比成生活中的东西。比如跟非技术的人解释缓存，我会说"就像你把常用的文件放在桌面上，不用每次都去柜子里翻"。多画图少说话，一张架构图或流程图比说十分钟有效。

## 3. 你工作中遇到的最大的一个非技术类挑战是什么？
最大的非技术挑战是在天猫好房做商品中台重构时的跨团队协调。
这个项目涉及十几个上下游团队，导购、搜索、交易、营销、详情都要配合改造接口。最难的不是技术方案，而是让这么多团队在同一个时间窗口内完成各自的改造。每个团队都有自己的业务排期，凭什么优先配合你？
我的做法是先找到各团队的利益点。比如搜索团队一直抱怨商品数据格式不统一导致索引效率低，我就跟他们说中台重构后数据格式标准化，他们的索引构建效率能提升30%。交易团队担心改造影响稳定性，我就承诺新老系统并行跑，灰度切流他们可以随时回滚。把"帮我改"变成"对你也有好处"。
然后拉了一个跨团队的周会，每周同步各方进度和风险，有卡点当场解决。遇到排期冲突的，升级到各团队TL层面对齐优先级。
最后项目按期上线了，这个过程让我意识到，越往上走，协调推动的能力比写代码重要得多。

## 4. 举一个能体现你沟通能力的例子
在天猫好房做商品体系升级的时候，产品提了一个需求，要把房产商品的属性体系从扁平结构改成树形结构，支持多级类目属性继承。产品觉得很简单，就是加几个层级嘛。
但我评估下来发现这个改动影响面很大，搜索、详情、营销三个团队的数据消费方式都要改，而且老数据迁移有兼容性风险。如果直接说"做不了"或者"要三个月"，产品肯定不接受。
我的做法是先约了产品单独聊，不是在大会上当面否定他。我画了一张数据流图，用产品能理解的方式解释了改动的影响范围，然后提了一个分阶段方案：第一期先做两级类目支持核心场景，两周上线；第二期再做完整的多级继承，排到下个迭代。
产品一看第一期就能满足80%的业务场景，马上就同意了。搜索和详情团队也松了口气，因为第一期他们只需要做很小的适配。
这个事情让我体会到，好的沟通不是说服对方接受你的方案，而是找到双方都能接受的最优解。

## 5. 举一个体现自己抗压能力的例子
天猫好房双11大促前两周，商品中台刚完成重构切流，突然发现一个数据一致性问题——部分房源的价格在新老系统之间有偏差，影响了大概5%的商品。
当时压力非常大，一方面大促倒计时不能延期，另一方面价格错了是P0级别的事故，直接影响交易。团队里有人开始慌了，觉得要不要回滚到老系统。
我先稳住团队情绪，跟大家说"问题可控，我们一个一个解决"。然后快速定位了根因——是数据迁移脚本在处理多SKU房源时有个边界case没覆盖到。我带着两个同学连续两天加班，写了修复脚本+全量对账工具，把所有有问题的数据修复了，同时加了实时对账监控防止再出现。
大促当天系统零故障，扛住了峰值流量。事后复盘的时候我主动承认了迁移方案的疏漏，把对账工具沉淀成了标准流程，后面每次数据变更都跑一遍对账。
那段时间确实很累，但扛过去之后团队的信心和凝聚力反而更强了。

## 6. 在海康做的比较成功的项目。
在海康做的最成功的是综合安防平台的停车场子系统，这是海康总部园区的海豚试点项目。
背景是海康园区原来的停车系统是各个子系统独立运行的，车牌识别、计费、车位引导、门禁各自一套，数据不通，运维成本高，用户体验也差。我负责把这些整合成一个统一的停车场子系统，接入到综合安防平台里。
最大的挑战是要对接十几种不同型号的硬件设备，每种设备的协议和数据格式都不一样。我设计了一个设备适配层，用策略模式封装不同设备的通信协议，上层业务逻辑完全不用关心底层是什么设备。这样后面新增设备型号只需要加一个适配器，不用改业务代码。
项目上线后在海康总部园区跑了半年，车辆通行效率提升了40%，无感支付覆盖率从0做到了85%。后来这套方案被推广到了海康的其他客户项目里，变成了一个标准化的产品模块。
这个项目让我第一次体会到做平台化产品的成就感，不是做完就完了，而是能被复用、能产生持续价值。

## 7. 你在原公司晋升了以后，整体的工作安排有变化吗?
变化挺大的，最明显的是从"自己干"变成了"带着团队干"。
晋升前我主要是核心开发，自己写代码、做方案、解决技术难题。晋升后角色转变了，更多的精力放在三件事上：
第一是架构把控。不再是每个需求都自己写代码了，而是做技术方案评审，把控系统的整体架构方向。比如在天猫好房，商品中台重构的整体方案是我设计的，但具体模块的开发分给了团队成员，我负责CR和关键节点的技术决策。
第二是团队培养。开始有意识地带人，把一些有挑战性的任务分给有潜力的同学，让他们在实战中成长。比如批处理框架这个项目，我让一个P6的同学主导开发，我做技术指导和方案把关，最后他独立完成了，这个项目也成了他晋升的核心素材。
第三是跨团队协调。晋升后参与的跨团队项目多了，需要跟产品、运营、上下游团队对齐目标和排期。沟通协调的时间占比从之前的10%涨到了30%左右。
总结就是从个人贡献者变成了团队的技术引领者，产出方式从"我做了什么"变成了"我带着团队做了什么"。
