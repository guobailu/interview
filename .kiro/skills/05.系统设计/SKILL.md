# 系统设计技能

## 触发条件
当用户提出系统设计相关问题时激活，包括但不限于：系统设计、模型设计、类图设计、数据库设计、架构设计等。

## 回答流程（必须严格遵守）
1. 收到系统设计题目后，不要直接作答，先针对题目提出3-5个关键问题，用于澄清业务场景、性能指标、边界约束等
2. 等待用户回答提问后，再结合用户的回答按照下面的设计背景和要求进行完整的系统设计作答
3. 如果用户回答比较模糊，可以追问一轮，但最多追问一次

## 提问方向参考
- 业务场景：核心业务流程是什么，有哪些关键角色
- 性能指标：预期QPS、数据量级、响应时间要求
- 边界约束：是否有技术栈限制、是否需要兼容已有系统
- 一致性要求：强一致还是最终一致，哪些环节不能出错
- 扩展性：未来是否有国际化、多租户、多业务线扩展需求

## 设计背景
基于符合分布式集群下支持百万QPS、海量数据的高可用、高性能、高稳定性、高并发、高负载、高扩展的现代企业级应用背景，以Java架构师的角度做系统设计。

## PlantUML类图要求
- 直接给出高并发下具体详细的复杂模型设计
- 图代码中必须包括入口到出口的所有交互、所有类、数据库设计（包括关联的外键）、每个属性字段设计（包括关联的外键）、所有方法及方法注释、方法的入参、出参等
- 图代码中属性字段一定要全，类一定要全，尽量详细而非简化版本
- 要求详细版本，不要写简化版本，不要写简单版本，直接写详细版本
- 每个类中必须添加版本号字段（version）
- 方法的入参也要添加版本号
- 适当添加设计模式、策略模式、优先级、调度算法、批处理、流式处理等字段
- 类至少包含7个
- 类之间的关系至少超过8个，需要体现业务流转
- 类之间的关系注释使用中文表述
- 类之间的线包含箭头
- 类之间的关系不要是发散的，尽量是环状，能体现整体的业务流转
- 类之间的关系有依赖、聚合、组合等
- 不允许存在数据库类（如DatabaseService、DataBaseService、Database等仅代表数据库的类）
- 必须体现数据库每张表的字段设计
- 数据库的实体类中的字段要列举完全
- 类名、方法名为英文，字段名为中文，不要加引号
- 尽量不要使用package标签
- 不要把箭头写进类关系的注释里，箭头应该在与--连接线相邻的左右两边上

## 方法注释格式
```
+ findTaskById(taskId: String): Task 根据ID查找任务
```

## PlantUML图代码后补充
在PlantUML图代码后面，将每个类的类注释补充进去。

## 技术选型与架构要求
根据具体业务场景结合进行技术选型，给出系统架构，包括：
- 负载均衡
- 关系型数据库、分库分表、读写分离
- 非关系型数据库：HBase、MongoDB、OpenSearch、ES
- 缓存、CDN
- 异步处理：RocketMQ、Kafka
- 大数据技术：Spark、Hadoop、HDFS
- 人工智能AI大模型
- 云计算、云原生
- 阿里开源分布式解决方案
- 领域驱动设计DDD思想
- 乐观锁、业务幂等、状态机流转、分布式事务、数据一致性
